## 2. 产品定义

### 2.1 核心定义

> **Atlax 是一个本地优先的知识整理与结构化工作台。**
> 
> 它帮助用户把零散输入、想法、片段记录、文档内容逐步沉淀为可分类、可关联、可回看的知识资产。
> 
> 第一阶段核心不是"替用户全自动思考"，而是通过清晰的交互、轻量规则引擎、标签体系、结构化视图，降低用户整理成本。

### 2.2 一句话定义

**降低整理成本，让知识沉淀更轻松。**

### 2.3 明确要点

| 要点 | 说明 |
|------|------|
| 输入源 | 手动输入为主，导入能力后续接入 |
| Tag 与 Suggestion 并存 | 用户主动选择 Tag：优先按用户意图归类；用户不选：系统给出建议 |
| 沉淀目标 | 不是"无限列表"，而是"有组织的知识结构" |
| 数据结构 | 树状感是用户体验目标，不是底层存储约束 |
| AI 定位 | AI 是增强项，不是产品成立前提 |

### 2.4 产品定位

Atlax 是：
- 本地优先的知识整理与结构化工作台
- 以"输入 → 整理 → 结构化沉淀 → 回看/利用"为主线
- 当前阶段不依赖 LLM 才能成立
- 通过 Tag 体系、规则引擎、结构化视图降低整理成本

Atlax 不是：
- 另一个 Obsidian（不做复杂双向链接系统）
- 另一个 Notion（不做重型数据库系统）
- AI 知识助手（不依赖大模型才能成立）
- 自动知识中台（不做全自动整理）

## 3. 为什么 Atlax 必须存在

### 3.1 现有工具没有解决什么？

| 问题 | 说明 | 用户痛点 |
|------|------|----------|
| 整理成本 | 记录后需要手动分类、打标签、建链接 | "记完就忘，整理太累" |
| 长期维护成本 | tag越来越乱、文件夹越来越杂 | "知识库越做越难用" |
| 知识激活问题 | 内容沉没、长期未复用、找不到 | "记了很多，但用不上" |
| 工具迁移成本 | Notion→Obsidian搬运痛苦 | "换工具等于重新开始" |

### 3.2 Atlax 解决的核心命题

Atlax 不是"更好的笔记软件"，而是：

**一个降低整理成本、让知识沉淀更轻松的工作台。**

核心价值：
- 不强迫用户先学会管理，才能开始积累知识
- 通过 Tag 体系降低分类成本
- 通过规则引擎给出可解释的建议
- 通过结构化视图让内容可管理、可回看

## 5. 核心主线功能

### 5.1 主线流程

```text
Capture（快速输入）
  → Inbox（查看待整理内容）
  → Suggest / Tag（系统建议 or 用户手动标记）
  → Archive / Organize（归档到结构化知识单元）
  → Browse / Review（回看、筛选、关系查看）
```

### 5.2 主线功能定义

| 功能 | 目标 | 说明 |
|------|------|------|
| **Capture** | 快速输入内容 | 文本输入、Quick Input Bar、不强制分类 |
| **Inbox** | 查看待整理内容 | 列表展示、状态流转（pending → suggested → archived） |
| **Suggest** | 系统给出建议 | 规则引擎、关键词匹配、上下文召回、可解释 |
| **Tag** | 用户手动标记 | Tag 选择器、Tag 创建、用户意图优先 |
| **Archive** | 归档到结构化单元 | Entry 创建、Tag 关联、项目归属 |
| **Browse** | 回看与筛选 | Entries 列表、基础筛选、多维度浏览 |
| **Review** | 回顾与激活 | 基础统计、健康信号、提醒卡片 |

## 6. Tag 正式能力

### 6.1 Tag 定位

Tag 是 Atlax 的基础能力，不是可有可无的小点缀。

| 定位 | 说明 |
|------|------|
| 分类服务 | Tag 服务内容分类，降低整理成本 |
| 检索服务 | Tag 服务后续检索、聚合 |
| 导入映射 | Tag 服务未来导入能力的数据映射 |
| 关系可视化 | Tag 服务关系可视化、关联发现 |

### 6.2 Tag 与 Suggestion 并存原则

```text
用户主动选择 Tag → 优先按用户意图归类
用户不选 Tag → 系统给出建议
用户接受建议 → 按建议归类
用户修正建议 → 按用户修正归类
```

### 6.3 Tag 是当前不接 LLM 时的关键减压设计

- Tag 让用户可以主动控制分类
- Tag 规则引擎可以给出可解释的建议
- Tag 不依赖大模型才能成立
- Tag 是产品成立的前提能力

## 7. Demo 界面目标

### 7.1 设计原则

| 原则 | 说明 |
|------|------|
| 不追求精致视觉稿 | 功能优先，体验其次 |
| 必须像真正的生产力工具 | 不是只有输入框的原型 |
| 尽量单页完成主要操作 | 减少频繁路由跳转 |

### 7.2 交互形态参考

参考 Codex Desktop / Notion：

```text
左侧导航或对象列表
右侧内容预览/整理面板
底部或局部输入区
```

### 7.3 工作台式 UI 结构

| 区域 | 功能 |
|------|------|
| Sidebar | 导航、对象列表、树形展开 |
| Content Panel | 内容预览、整理面板、详情展示 |
| Quick Input Bar | 快速输入、常驻输入框 |
| Action Area | 统一操作区（归档、删除、编辑等） |

## 8. 导入能力：Deferred but Required

### 8.1 战略定位

导入能力是产品长期关键能力，但当前 Demo 阶段不交付完整导入。

### 8.2 当前阶段

- 不实现完整导入
- 预留导入入口
- 预留 importer interface

### 8.3 未来规划

导入能力需要：
- 对外部 Markdown/文档导入后，能自动进入 Inbox
- 导入内容能进入 tag/suggestion/归档流程
- 导入内容能映射到现有 Tag 体系

## 12. 数据结构原则

### 12.1 树状感是用户体验目标，不是底层存储约束

| 层级 | 说明 |
|------|------|
| 用户体验层 | 可以提供"像树一样有层次感"的组织体验 |
| 底层数据模型 | 采用适合查询、关联、过滤、扩展的结构化模型 |
| 不要求 | 不要求真实物理存储为树 |

### 12.2 数据模型原则

| 原则 | 说明 |
|------|------|
| 不绑定 UI 视图 | Domain model 不绑定 UI 视图 |
| 适合查询 | 不能使用只适合展示、不适合查询的数据结构 |
| 预留扩展 | 为后续 tag filter、relationship query、review aggregation 留空间 |

## 13. 验收标准

### 13.1 Phase 2 Demo 验收

| 项目 | 验收标准 |
|------|----------|
| Capture | 用户可在 10 秒内完成新建内容并进入 Inbox |
| Inbox | 每条内容都能看到状态、建议、Tag 选项 |
| Suggest | 系统能给出可解释的建议 |
| Tag | 用户能主动选择 Tag、创建 Tag |
| Archive | 用户能完成归档，内容进入 Entries |
| Browse | 用户能在列表中找到归档内容 |
| UI | 像真正的生产力工具，不是只有输入框的原型 |

### 13.2 产品验收

Phase 2 Demo 通过的判断：

- 用户能快速记录，不被结构打断
- 用户会觉得整理比自己手动做更轻松
- Demo 像真正的生产力工具

## 14. 版本差异说明（v4.0 相对旧版）

本次收敛更新的核心变化：

1. **产品定位收敛**：从"Personal Knowledge OS"改为"本地优先的知识整理与结构化工作台"
2. **核心主线明确**：Capture → Inbox → Suggest/Tag → Archive → Browse/Review
3. **Tag 正式能力**：明确 Tag 是基础能力，不是可有可无的小点缀
4. **导入能力定位**：明确"deferred but required"，预留接口
5. **非目标明确**：明确不做重型任务管理器、协作系统、依赖大模型的中台
6. **Phase 规划统一**：使用 Phase 1-4 语言，不再使用 Week 语言
7. **数据结构原则**：明确"树状感是用户体验目标，不是底层存储约束"
