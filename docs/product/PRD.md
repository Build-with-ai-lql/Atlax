## 2. 产品定义

### 2.1 核心定义

> **Atlax MindDock 是一个本地优先、账号驱动的知识整理与结构化工作台。**
> 
> 它帮助用户把零散输入、想法、片段记录、文档内容逐步沉淀为可分类、可关联、可回看的知识资产。
> 
> 第一阶段核心不是"替用户全自动思考"，而是通过清晰的交互、轻量规则引擎、Tag 体系、结构化视图，降低用户整理成本。
>
> 本地优先不等于完全离线孤岛：整理、推荐与隐私敏感分析优先在本地完成，但 Demo 阶段就需要可用的账号体系；云同步与协作能力可延后交付，但接口与授权边界必须预留，为多端使用与云存储接入打基础。

### 2.2 一句话定义

**降低整理成本，让知识沉淀更轻松。**

### 2.3 明确要点

| 要点 | 说明 |
|------|------|
| 输入源 | 手动输入为主，导入能力后续接入 |
| Tag 与 Suggestion 并存 | 用户主动选择 Tag 时始终以用户意图为准；用户不选时系统仅提供低打扰、可解释、可修正的建议，不默认把用户拖入高学习成本的双向链接交互 |
| Dock 工作台 | 所有待整理内容先进入 Dock，而不是直接要求用户建立复杂结构 |
| 沉淀目标 | 不是"无限列表"，而是"有组织的知识结构" |
| 数据结构 | 树状感是用户体验目标，不是底层存储约束 |
| AI 定位 | AI 是增强项，不是产品成立前提 |

### 2.4 产品定位

Atlax 是：
- 本地优先的知识整理与结构化工作台
- 本地处理优先，且 Demo 阶段就必须有账号体系承载用户身份与后续扩展基础
- 以"输入 → 整理 → 结构化沉淀 → 回看/利用"为主线
- 当前阶段不依赖 LLM 才能成立
- 通过 Tag 体系、规则引擎、结构化视图降低整理成本

Atlax 不是：
- 另一个 Obsidian（不直接复用高学习成本的双向链接逻辑）
- 另一个 Notion（不做重型数据库系统）
- AI 知识助手（不是必须依赖大模型才可用）
- 自动知识中台（不默认做黑盒式全自动整理）

## 3. 为什么 Atlax 必须存在

### 3.1 现有工具没有解决什么？

| 问题 | 说明 | 用户痛点 |
|------|------|----------|
| 整理成本 | 记录后需要手动分类、打标签、建链接 | "记完就忘，整理太累" |
| 长期维护成本 | tag越来越乱、文件夹越来越杂 | "知识库越做越难用" |
| 知识激活问题 | 内容沉没、长期未复用、找不到 | "记了很多，但用不上" |
| 工具迁移成本 | Notion→Obsidian搬运痛苦 | "换工具等于重新开始" |

### 3.2 Atlax 解决的核心命题

Atlax 不是"更好的笔记软件"，而是：

**一个降低整理成本、让知识沉淀更轻松的工作台。**

核心价值：
- 不强迫用户先学会管理，才能开始积累知识
- 通过 Tag 体系降低分类成本
- 通过可信的推荐能力给出可解释、可修正的建议
- 通过结构化视图让内容可管理、可回看

## 5. 核心主线功能

### 5.1 主线流程

```text
Capture（快速输入）
  → Dock（查看待整理内容）
  → Suggest / Tag（系统建议 or 用户手动标记）
  → Archive / Organize（归档到结构化知识单元）
  → Browse / Review（回看、筛选、关系查看）
  → Re-organize（需要时返回 Dock 继续编辑与重整）
```

### 5.2 主线功能定义

| 功能 | 目标 | 说明 |
|------|------|------|
| **Capture** | 快速输入内容 | 文本输入、Quick Input Bar、不强制分类 |
| **Dock** | 查看待整理内容 | 列表展示、状态流转（pending → suggested → archived），是 MindDock 的统一待整理入口 |
| **Suggest** | 系统给出建议 | 规则引擎、关键词匹配、历史行为信号、结构信号、可解释 |
| **Tag** | 用户手动标记 | Tag 选择器、Tag 创建、用户意图优先 |
| **Archive** | 归档到结构化单元 | Entry 创建、Tag 关联、项目归属；归档后仍可重新打开并回到 Dock 再整理 |
| **Browse** | 回看与筛选 | Entries 列表、基础筛选、多维度浏览、关系查看入口 |
| **Review** | 回顾与激活 | 基础统计、健康信号、提醒卡片、归档结构健康度提示 |

### 5.3 归档不是一次性终局

- 已归档内容必须支持再次编辑、补充 Tag、调整归属、重新整理。
- Browse / Review 不是只读结果页，而是持续整理流程中的回看与再组织入口。
- 产品不能把用户锁死在"一次性纳入"流程里，否则无法支撑真实知识沉淀。

## 6. Tag 正式能力

### 6.1 Tag 定位

Tag 是 Atlax 的基础能力，不是可有可无的小点缀。

| 定位 | 说明 |
|------|------|
| 分类服务 | Tag 服务内容分类，降低整理成本 |
| 检索服务 | Tag 服务后续检索、聚合 |
| 导入映射 | Tag 服务未来导入能力的数据映射 |
| 关系可视化 | Tag 服务关系可视化、关联发现；双向 Tag / 关系链接可作为增强能力，但默认以清晰视图与推荐为主，不直接暴露高学习成本的复杂编辑器 |

### 6.2 Tag 与 Suggestion 并存原则

```text
用户主动选择 Tag → 始终优先按用户意图归类
用户不选 Tag → 系统给出低打扰建议，而不是强制替用户决定
用户接受建议 → 按建议归类，并记录为后续偏好信号
用户修正建议 → 以用户修正结果为准，并用于优化后续建议
```

### 6.3 可信推荐原则

- 推荐质量直接影响用户对产品的信任，错误建议不能长期重复出现。
- 推荐优先依赖本地算法与本地信号，避免默认上传原始内容。
- 推荐信号至少包括：用户历史 Tag 选择、Tag 链路与关系链路、归档路径、内容结构特征、用户对建议的接受/修正反馈。
- 推荐必须可解释，至少要让用户理解"为什么会推荐这个 Tag / 关系"。
- 推荐必须可关闭、可忽略、可修正，不能把错误建议变成新的使用负担。

### 6.4 Tag 是当前不接 LLM 时的关键减压设计

- Tag 让用户可以主动控制分类
- Tag 规则引擎可以给出可解释的建议
- Tag 不依赖大模型才能成立
- Tag 是产品成立的前提能力

## 7. Demo 界面目标

### 7.1 设计原则

| 原则 | 说明 |
|------|------|
| 不追求精致视觉稿 | 功能优先，体验其次 |
| 必须像真正的生产力工具 | 不是只有输入框的原型 |
| 尽量单页完成主要操作 | 减少频繁路由跳转 |
| 输入能力必须覆盖长短内容 | 不能只提供一个适合短句的记录框，长笔记 / 长文章也必须有合理入口 |
| 输入区不能破坏主工作台结构 | 输入组件不应横跨两个模块，避免版式割裂与阅读层级混乱 |

### 7.2 交互形态参考

参考 Codex Desktop / Notion：

```text
左侧导航或对象列表
右侧内容预览/整理面板
底部局部快速输入区
可展开的大编辑器 / 富文本输入层
```

### 7.3 工作台式 UI 结构

| 区域 | 功能 |
|------|------|
| Sidebar | 导航、对象列表、树形展开 |
| Content Panel | 内容预览、整理面板、详情展示 |
| Quick Input Bar | 快速输入、灵感记录、短文本捕获 |
| Expanded Editor | 面向长笔记 / 长文章 / 富文本输入的大编辑器，可由快速输入区展开进入 |
| Action Area | 统一操作区（归档、删除、编辑等） |

### 7.4 Capture 输入交互原则

- Capture 不能只有单一的小输入框。产品既要支持灵感、短句、待办，也要支持较长笔记、整篇文章与高信息密度记录。
- Phase 2 需要建立双形态输入：
  - 快速输入形态：面向灵感、短文本、临时记录，保持低门槛。
  - 展开输入形态：面向长内容、结构化笔记与富文本编辑，支持更大的编辑空间。
- 快速输入区应提供明确的扩展动作，例如加号按钮、展开按钮或独立入口；用户进入展开态后，应看到更接近飞书 / 富文本编辑器的输入体验。
- 快速输入区与展开编辑器应属于同一条 Capture 主线，而不是两个割裂功能。
- 输入区布局必须服从主工作台的信息架构，不能为了常驻输入而横跨两个模块，破坏列表区与详情区的结构边界。
- Phase 2 先解决“能不能合理输入长短内容”和“布局是否成立”；Phase 3 再处理富文本细节、动画反馈、工具栏层级与精细交互。

## 8. 导入能力：Deferred but Required

### 8.1 战略定位

导入能力是产品长期关键能力，但当前 Demo 阶段不交付完整导入。

### 8.2 当前阶段

- 不实现完整导入
- 预留导入入口
- 预留 importer interface

### 8.3 未来规划

导入能力需要：
- 对外部 Markdown/文档导入后，能自动进入 Dock
- 导入内容能进入 tag/suggestion/归档流程
- 导入内容能映射到现有 Tag 体系

## 9. 当前非目标与延后能力

### 9.1 明确不做

| 非目标 | 说明 |
|------|------|
| 重型任务管理器 | 当前不做复杂任务系统、GTD 系统，但必须支持归档内容的持续整理与结构维护，并提供足够的算法能力做 Tag 链路与关系推荐 |
| 复杂协作系统 | 当前不做多人实时协作、细粒度权限系统，但必须预留账号授权、同步与后续协作扩展位 |
| 依赖大模型的自动知识中台 | 产品不依赖 LLM 才能成立，但要允许本地算法推荐与后续 AI provider 接入，支持用户自带模型或平台内置模型 |
| 过度复杂的图谱编辑器 | 当前不做复杂图谱编辑器或 Obsidian 式双向链接重度交互，但必须支持清晰的关系查看、Tag 链路推荐与后续结构整理 |

### 9.2 延后但保留接口

| 延后项 | 说明 |
|------|------|
| 完整导入能力 | Phase 4，预留 importer interface |
| 语音输入与扩展采集 | Phase 4，作为扩展输入能力推进 |
| 图谱可视化增强 | Phase 4，预留 graph view 扩展点，保证视图清晰可控 |
| 复杂关系计算 | Phase 4，预留 relation 计算接口 |
| 搜索增强 | Phase 4，预留索引设计 |
| 高级 Review / Insight | Phase 4，作为结构激活增强能力推进 |
| 账号登录形态扩展 | Phase 5，支持 Apple / Google / GitHub / 微信 / X / 手机号等登录形态扩展 |
| 云同步能力 | Phase 5，预留 iCloud、Google Drive、OneDrive 等存储接入位 |
| AI 增强建议 | Phase 5，预留 AI provider 接口，支持用户自带模型或平台内置模型（例如 DeepSeek） |
| 多人协作 | Phase 5，在用户授权前提下逐步开放 |

### 9.3 必须预留的基础设施

- 账号系统不是可选项。Demo 阶段就需要登录后才能管理个人资料、身份凭证与后续同步授权能力。
- 本地优先不等于服务端持有全部内容；隐私策略应以本地处理、最小上传、用户授权同步为原则。
- 协作不是当前交付重点，但数据模型、权限边界、共享链接与授权流程必须从架构上留接口。
- AI 不是当前主线，但推荐链路、provider 接口、计费入口与模型切换策略需要提前设计；默认优先本地算法，外部模型作为增强层接入。

## 10. Phase 规划

### 10.1 Phase 1 / MVP 闭环

**状态**：已完成或接近完成

| 内容 | 说明 |
|------|------|
| 输入 | Capture 基础能力 |
| 本地闭环 | 本地存储、基础结构 |
| 基础结构 | Entry、Tag 基础模型 |

### 10.2 Phase 2 / Demo 闭环与可用基线

**状态**：当前阶段

| 内容 | 说明 |
|------|------|
| 完整主线功能补齐 | Capture → Dock → Suggest/Tag → Archive → Browse/Review → Re-organize |
| 账号系统成立 | 用户可注册 / 登录 / 退出，识别身份并进入自己的工作区 |
| 用户基础管理可用 | 基础账号状态、会话保持、用户资料与工作区入口成立 |
| 工作台总览成立 | 用户进入总工作台后能看到自己的知识库总览与清晰结构入口 |
| 工作台式界面成型 | Sidebar + Content Panel + Quick Input Bar |
| Capture 双形态输入成立 | 短输入保留低门槛入口，长内容通过展开编辑器进入，不再被单一小输入框限制 |
| 输入区布局完成结构纠偏 | 输入组件不横跨两个主模块，主工作台排版边界清晰 |
| Suggestion + Tag 并存 | 用户主动选择优先，系统建议补充，且建议可修正 |
| 数据库 / 知识库浏览可用 | 已归档内容可被查看、筛选、打开、编辑，而不是仅能演示 |
| 归档后二次整理成立 | 已归档内容可重新编辑、补 Tag、调整归属并回到 Dock |
| Review 基础版成立 | 用户可看到最小统计、健康信号与知识激活入口 |
| Demo 可用而不只是可操作 | 核心功能具备稳定可用性，能支持测试之外的真实试用 |

### 10.3 Phase 3 / 产品化打磨与前端交互优化

**状态**：后续阶段

**启动前提**：只有当 Phase 2 的主线功能已经全部闭环且达到“可用”标准后，才能进入 Phase 3。Phase 3 不是替未完成的功能补洞，而是基于稳定能力做产品化打磨。

| 内容 | 说明 |
|------|------|
| 核心价值表达统一 | 首页、空状态、模块引导、总工作台信息架构与主 CTA 统一 |
| Capture 与 Dock 体验打磨 | 输入反馈、富文本工具栏层级、展开态切换、建议区、操作层级、状态表达与错误恢复优化 |
| Browse / Review / Database 展示优化 | 浏览、回看、统计、数据库字段层级与关系入口的阅读顺序更清晰 |
| 工作台总览与知识库可读性提升 | 总览页、知识库概览、对象列表与详情区形成清晰阅读结构 |
| 用户管理与账户交互打磨 | 登录、注册、会话状态、设置入口、权限提示与用户反馈更完整 |
| 全局交互反馈统一 | 按钮层级、加载/成功/失败状态、空状态、基础动效、响应式与可访问性统一 |
| 初具模型的成品化表达 | 阶段结束后应呈现出更完整、可被真实使用的初具模型成品，而不是“大一点的 Demo” |

### 10.4 Phase 4 / 扩展功能与结构增强

## 12. 数据结构原则

### 12.1 树状感是用户体验目标，不是底层存储约束

| 层级 | 说明 |
|------|------|
| 用户体验层 | 可以提供"像树一样有层次感"的组织体验 |
| 底层数据模型 | 采用适合查询、关联、过滤、扩展的结构化模型 |
| 不要求 | 不要求真实物理存储为树 |

### 12.2 数据模型原则

| 原则 | 说明 |
|------|------|
| 不绑定 UI 视图 | Domain model 不绑定 UI 视图 |
| 适合查询 | 不能使用只适合展示、不适合查询的数据结构 |
| 预留扩展 | 为后续 tag filter、relationship query、review aggregation 留空间 |

## 13. 验收标准

### 13.1 Phase 2 Demo 验收

| 项目 | 验收标准 |
|------|----------|
| Capture | 用户可在 10 秒内完成短内容新建并进入 Dock；长内容也有明确展开入口，不被单一小输入框限制 |
| Account | 用户必须能登录并进入自己的工作区，身份状态在刷新后可保持 |
| Workspace | 用户进入总工作台后，能够看到自己的知识库总览，且结构入口清晰可读 |
| Dock | 每条内容都能看到状态、建议、Tag 选项 |
| Suggest | 系统能给出可解释的建议，且用户可忽略或修正 |
| Tag | 用户能主动选择 Tag、创建 Tag |
| Archive | 用户能完成归档，内容进入 Entries，且可重新打开继续整理 |
| Browse / Database | 用户能在列表中找到归档内容，完成查看、筛选、打开与再次整理 |
| Review | 用户能看到基础统计、健康信号与知识激活入口，而不是空壳页面 |
| UI | 像真正的生产力工具，不是只有输入框的原型；输入区不横跨两个模块，主工作台排版边界清晰 |

### 13.2 产品验收

Phase 2 Demo 通过的判断：

- 用户能快速记录，不被结构打断
- 用户会觉得整理比自己手动做更轻松，而不是被错误建议拖慢
- Demo 像真正的生产力工具

## 14. 版本差异说明（v4.7 相对 v4.6）

本次补充更新的核心变化：

1. **Capture 输入从单一记录框升级为双形态要求**：Phase 2 必须同时支持短输入与长内容输入，不能再只适配短句记录
2. **输入区布局约束被正式写入 PRD**：输入组件不能横跨两个主模块，必须服从主工作台信息架构
3. **阶段边界进一步收紧**：Phase 2 先解决输入能力与结构问题，Phase 3 再承接富文本细节和前端精细交互打磨
