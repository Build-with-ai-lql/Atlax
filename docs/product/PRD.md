## 2. 为什么 Atlax 必须存在

> **这是 Atlax 存在的根本理由。**

### 2.1 现有工具解决了什么？

现有知识工具解决了**记录问题**：
- Obsidian：本地化、Markdown、双向链接
- Notion：数据库视图、all-in-one、协作
- Apple Notes：快速捕捉、系统集成
- Logseq：大纲式、每日笔记

### 2.2 现有工具没有解决什么？

| 问题 | 说明 | 用户痛点 |
|------|------|----------|
| **整理成本** | 记录后需要手动分类、打标签、建链接 | "记完就忘，整理太累" |
| **长期维护成本** | tag越来越乱、文件夹越来越杂、链接越来越噪 | "知识库越做越难用" |
| **知识激活问题** | 内容沉没、长期未复用、找不到 | "记了很多，但用不上" |
| **工具迁移成本** | Notion→Obsidian搬运痛苦、数据锁定 | "换工具等于重新开始" |
| **人生项目管理问题** | 项目停滞、目标遗忘、进度不可见 | "不知道自己在做什么" |

### 2.3 Atlax 解决的核心命题

> **用户买的不是工具，是被解决的人生问题。**

Atlax 不是"更好的笔记软件"，而是：

**一个会主动帮你推进人生与工作的知识系统。**

## 3. 核心爆点定位

> **让用户一句话记住 Atlax。**

### 3.1 一句话定位（市场记忆点）

# **Atlax：让知识开始替你工作**

副标题：

> **输入任何内容，系统自动整理、关联、复盘、推动行动。**

### 3.2 为什么这个定位更强？

| 维度 | 说明 |
|------|------|
| **价值导向** | 用户真正买单的是：少花时间整理、找得回来、能推进人生和工作 |
| **差异化明显** | 不是"自动整理"四个字本身，而是"知识替你工作"的体验 |
| **记忆成本低** | 一句话就能让用户记住产品定位 |
| **传播性强** | 用户可以一句话向朋友介绍："有个产品会让知识替你工作" |

### 3.3 核心爆点验证闭环

```
用户输入任意内容 → 系统自动进入Inbox → 系统给出分类建议 →

```

**这个闭环成立，产品价值就成立。**

## 4. 产品概述

### 4.1 产品层级定义

# **Atlax 不是笔记软件，而是 Personal Knowledge OS（个人知识操作系统）。**

它负责：
- **输入**：捕捉一切灵感、想法、信息
- **整理**：自动分类、打标签、归档
- **连接**：建立知识网络、发现关联
- **回顾**：定期复盘、激活沉没内容
- **推动行动**：项目进度提醒、目标追踪

### 4.2 一句话定义
**Atlax 是一款让知识开始替你工作的个人知识操作系统。**

### 4.3 核心价值主张
它不是一个"功能更全的笔记软件"，而是一个让用户**不必先学会管理，才能开始积累知识**的系统。

核心不在于"支持多少输入方式"，而在于：**不管用户怎么输入，系统都能先接住内容，再帮助用户完成第一轮归类、结构化和后续整理，并主动推动用户前进。**

### 4.4 产品定位
Atlax 是一款：
- **个人知识操作系统**（Personal Knowledge OS）
- 本地优先的个人知识工作流工具
- 面向高信息密度知识工作者的结构化笔记系统
- 支持多输入方式的知识捕捉与归类平台
- 以 Markdown 为长期资产层、以 Database 为组织层、以 Insight Engine 为推动层的系统

Atlax 不是：
- 另一个 Obsidian
- 另一个 Notion
- 另一个 GoodNotes
- 单纯的语音笔记软件
- 单纯的 Markdown 编辑器
- 单纯的 AI 摘要工具

## 5. 护城河分析

> **用户问：为什么不用 Notion + AI 插件？**
> **这是 Atlax 必须回答的核心问题。**

### 5.1 Atlax 护城河

| 护城河 | 说明 | 竞争对手难以复制的原因 |
|------|------|------------------------|
| **Personal Knowledge OS 定位** | 不是笔记软件，是知识操作系统 | 需要重新定义产品形态，现有工具无法转型 |
| **Local-first + Structured-native** | 本地优先 + 结构化原生 | Notion 无法本地化，Obsidian 无法结构化原生 |
| **输入到行动闭环** | 从输入到整理到推动行动的完整闭环 | 需要重新设计工作流，现有工具没有 Inbox 和 Insight Engine |
| **Knowledge Health 检查** | 知识库健康检查、主动提醒 | 市面上几乎没有产品做好这个功能 |
| **可迁移不锁死** | 支持完整导出，不锁死用户数据 | Notion 商业模式依赖数据锁定，不会主动开放 |

### 5.2 为什么现有工具无法解决？

| 工具 | 无法解决的问题 | Atlax 如何解决 |
|------|----------------|----------------|
| **Notion** | 本地资产弱、数据锁定、无主动提醒 | 本地优先存储、完整导出、Insight Engine |
| **Obsidian** | 上手难、无结构化视图、无健康检查 | Inbox自动整理、Database视图、Knowledge Health |
| **Apple Notes** | 无结构化能力、无知识激活 | 结构化原生、Review功能 |
| **Logseq** | 学习成本高、无项目推动 | 自由输入、Insight Engine |
| **Notion + AI插件** | AI只是附加功能，不是核心工作流 | Insight Engine 是核心层，不是插件 |

## 6. 目标用户（精准收窄版）

> **MVP 第一枪只打一个人群。**

### 6.1 第一批核心用户

# **高信息密度个人工作者**

| 用户类型 | 特征 | 为什么是第一批用户 |
|------|------|---------------------|
| **产品经理** | 高频记录、多项目并行、需要结构化管理 | 痛点最强，付费意愿高，传播能力强 |
| **程序员** | 技术敏感、喜欢本地化、Markdown原生用户 | 与Obsidian用户重叠，迁移成本低 |
| **创作者** | 内容生产者、需要知识复用、个人工作流 | 痛点明确，愿意尝试新工具 |
| **独立开发者** | 多项目、需要效率工具、技术敏感 | 痛点强，愿意折腾，会传播 |
| **内容创业者** | 高频输出、需要知识管理、付费意愿强 | 痛点明确，愿意付费 |

### 6.2 为什么不打学生？

| 原因 | 说明 |
|------|------|
| 付费意愿低 | 学生群体付费转化率低 |
| 信息密度低 | 学生笔记多为课堂记录，整理需求弱 |
| 传播价值低 | 学生圈层传播对产品品牌帮助有限 |
| 使用场景单一 | 学生场景不足以验证产品核心价值 |

### 6.3 用户特征（第一批用户）

| 特征 | 说明 |
|------|------|
| 有持续记录习惯 | 每天至少记录3条以上内容 |
| 有一定信息密度 | 内容有深度，不是流水账 |
| 知道整理重要，但没时间维护 | 痛点明确，有解决意愿 |
| 对工具体验敏感 | 愿意为更好的体验付费 |
| 希望知识能够长期复用 | 有知识资产意识 |
| 希望系统比自己更先接住内容 | 对"系统代整理"有需求 |
| **希望知识能推动自己前进** | **对"主动提醒"有需求** |

### 6.4 后续用户扩展计划

| 阶段 | 目标用户 | 说明 |
|------|----------|------|
| MVP | PM、程序员、创作者、独立开发者、内容创业者 | 验证核心价值 |
| Phase 2 | 设计师、研究者、自媒体 | 扩展到更多知识工作者 |
| Phase 3 | 学生、普通用户 | 产品成熟后下沉 |

## 8. 北极星指标

> **所有功能围绕一个核心指标设计。**

### 8.1 North Star Metric

**每周被整理并复用的内容数**

```
North Star = 每周进入 Database 的笔记数 × 被搜索/打开/关联的次数
```

### 8.2 为什么选择这个指标？

| 原因 | 说明 |
|------|------|
| **直接验证核心价值** | "知识替你工作"的价值在于整理后的内容被复用 |
| **用户行为导向** | 指标引导用户完成"输入→整理→复用"闭环 |
| **可量化可追踪** | 数据可直接从系统获取 |
| **与商业价值关联** | 复用越多，用户粘性越强，付费意愿越高 |

### 8.3 关键子指标

| 子指标 | 目标值（MVP） | 说明 |
|------|---------------|------|
| 新建笔记数/周 | > 10 | 用户有持续记录习惯 |
| Inbox转化率 | > 80% | 系统整理建议被接受 |
| 归档成功率 | > 90% | 用户完成整理闭环 |
| 搜索使用率 | > 50% | 用户主动检索复用 |
| 打开频率 | > 2次/笔记 | 内容被反复查看 |
| 关联密度 | > 1.5条/笔记 | 知识网络形成 |
| **Insight响应率** | > 30% | **用户响应主动提醒** |

### 9.3 核心模块详细设计

#### 9.3.1 Capture 模块（P0）

**目标**：让用户在最短时间内把内容丢进系统。

**支持的输入方式**：
| 类型 | MVP | 说明 |
|------|-----|------|
| 文本输入 | ✅ | 核心功能 |
| 语音输入 | ✅ | 快速捕捉 |

**入口形式**：
- 全局快捷键弹窗
- 新建按钮
- 快速语音记录按钮

**设计原则**：
- 可以无标题保存
- 可以不选分类保存
- 必须先进 Inbox
- 系统自动补建议

---

#### 9.3.2 Inbox AI 整理模块（P0）

**目标**：系统自动整理，用户一键确认。

**功能列表**：
| 功能 | MVP | 说明 |
|------|-----|------|
| 待处理列表 | ✅ | 显示所有未整理内容 |
| 自动类型建议 | ✅ | 系统建议笔记类型 |
| tag 建议 | ✅ | 系统建议标签 |
| project 建议 | ✅ | 系统建议关联项目 |
| 一键接受建议 | ✅ | 快速确认 |
| 一键归档 | ✅ | 快速归档 |
| 批量归档 | ✅ | 批量操作 |

**关键交互**：
- 左侧：待处理列表
- 中间：内容预览
- 右侧：建议面板
- 操作区：接受 / 调整 / 忽略 / 删除

---

#### 9.3.3 Database View 模块（P0）

**目标**：像 Notion 一样管理知识。

**支持对象**：
- Notes
- Projects
- Tasks

**视图支持**：
| 视图 | MVP | 说明 |
|------|-----|------|
| Table View | ✅ | 核心视图 |
| List View | ✅ | 简单列表 |

**基础字段 - Note**：
| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识 |
| title | string | 标题 |
| type | string | 类型 |
| status | string | 状态 |
| tags | array | 标签列表 |
| project_id | string | 关联项目 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

---

#### 9.3.4 Weekly Review 模块（P0）

**目标**：自动周报 + 健康检查，让用户看清人生进度。

#### 9.4.1 Knowledge Health 模块（P1）

**功能列表**：

#### 9.4.2 Insight Engine 模块（P1）

> **不是问答 AI，是主动提醒。**

**目标**：AI 主动提醒，让知识推动用户前进。

**功能列表**：
| 功能 | MVP | 说明 |
|------|-----|------|
| 项目停滞提醒 | ✅ | "支付系统项目 10 天无进展，是否需要回顾卡点？" |

| 知识激活提醒 | ✅ | "这篇笔记 30 天未打开，可能需要回顾" |
| 关联发现提醒 | ✅ | "发现 3 条笔记与当前内容相关" |

**设计原则**：
- 只做主动提醒，不做问答
- 提醒可关闭、可调整频率
- 提醒结果可一键执行

## 11. 产品原则

1. **输入自由，不强迫分类**
2. **系统先接住，再做整理**
3. **每条内容既是文档，也是数据库条目**
4. **Markdown 是长期资产层**
5. **Database 是组织层**
6. **自动化只做建议，不做强制接管**
7. **第一版优先稳定，不优先炫技**
8. **可导出、可迁移、不锁死**
9. **本地优先，云端能力后置**
10. **先做个人闭环，再谈协作**
11. **增长飞轮：让用户有可分享的内容**
12. **知识推动用户前进：Insight Engine 是核心层**（新增）

## 14. 风险评估

### 14.1 产品风险
| 风险 | 说明 | 应对策略 |
|------|------|----------|
| 容易做成四不像 | 功能边界不清 | MVP 只做 4 核心模块 |
| 功能边界不清 | 用户以为是另一个笔记软件 | 强调"Personal Knowledge OS"定位 |
| 用户理解偏差 | 用户不理解 Inbox 工作流 | 优化引导流程，提供示例 |
| 核心爆点不够炸 | 用户没有立刻下载的理由 | 强化"知识替你工作"定位 |
| 增长飞轮不转 | 用户不愿意分享 | 优化成长报告/项目周报设计 |

### 14.2 技术风险
| 风险 | 说明 | 应对策略 |
|------|------|----------|
| 自动归类效果不稳定 | 分类建议不准确 | 先做规则引擎，AI 后置 |
| Notion 导入映射复杂 | 数据结构差异大 | MVP 只做基础导入 |
| 双模型一致性难维护 | Markdown 与数据库同步 | 统一数据模型设计 |
| 双写一致性 | Markdown+DB同步问题 | 详细设计同步策略（见ARCHITECTURE） |
| 本地优先同步 | 多端数据同步复杂 | MVP不做同步，后续设计 |

### 14.3 开发风险
| 风险 | 说明 | 应对策略 |
|------|------|----------|
| 多端同时推进会失控 | 资源分散 | 先做 Web Demo |
| 手写太早上会拖慢节奏 | 技术复杂度高 | 手写后置 |
| AI 接太深会抬高成本 | 成本不可控 | AI 后置，规则优先 |
| 功能太多导致四不像 | MVP范围太宽 | MVP 只做 4 核心模块 |
