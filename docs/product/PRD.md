## 2. 产品定义

### 2.1 核心定义

> **Atlax MindDock 是一个本地优先、结构化优先、账号驱动的知识整理与结构化工作台。**
>
> 它同时支持经典工作台模式与对话式引导模式，帮助用户把零散输入、想法、片段记录、文档内容逐步沉淀为可分类、可关联、可回看的知识资产，并在不增加高昂整理成本的前提下降低表达启动成本。
>
> 第一阶段核心不是“替用户全自动思考”，而是通过清晰的交互、轻量规则引擎、Tag 体系、结构化视图与引导式对话能力，降低用户整理成本，让用户更容易开始表达、开始归类、开始沉淀。
>
> 本地优先不等于完全离线孤岛：整理、推荐与隐私敏感分析优先在本地完成，但 Demo 阶段就需要可用的账号体系；云同步与协作能力可延后交付，但接口与授权边界必须预留，为多端使用与用户自有云存储接入打基础。

### 2.2 一句话定义

**Atlax 是一个本地优先、结构化优先的知识管理工作台。它同时支持经典工作台模式与对话式引导模式，帮助用户在不增加高昂整理成本的前提下完成输入、归类、沉淀、回顾与再组织，并降低表达启动成本。**

### 2.3 明确要点

| 要点 | 说明 |
|------|------|
| 输入源 | 手动输入为主，导入能力后续接入 |
| 双模式交互 | Classic / Chat 只是交互入口不同，不是功能边界不同，服务同一知识库 |
| Dock 工作台 | 所有待整理内容先进入 Dock，而不是直接要求用户建立复杂结构 |
| Tag 与 Suggestion 并存 | 用户主动选择 Tag 时始终以用户意图为准；用户不选时系统提供低打扰、可解释、可修正的建议 |
| 沉淀目标 | 不是“无限列表”，而是“有组织的知识结构” |
| 数据结构 | 树状感是用户体验目标，不是底层存储约束 |
| AI 定位 | AI 是增强项，不是产品成立前提；Chat Mode 不是普通问答框，而是面向知识整理的引导式交互 |

### 2.4 产品定位

Atlax 是：
- 本地优先的知识整理与结构化工作台
- 同时提供经典工作台模式与对话式引导模式，但两者共用同一知识库、同一 Tag 体系、同一归档与回顾机制
- 以“输入 → 整理 → 结构化沉淀 → 回看/利用”为主线
- 当前阶段不依赖 LLM 才能成立
- 通过 Tag 体系、规则引擎、结构化视图与引导式对话降低整理成本与表达启动门槛

Atlax 不是：
- 另一个 Obsidian（不直接复用高学习成本的双向链接逻辑）
- 另一个 Notion（不做重型数据库系统）
- AI 知识助手（不是必须依赖大模型才可用）
- 自动知识中台（不默认做黑盒式全自动整理）
- 两套割裂的产品系统（Classic / Chat 不做双知识库、不做双主线）

## 3. 为什么 Atlax 必须存在

### 3.1 现有工具没有解决什么？

| 问题 | 说明 | 用户痛点 |
|------|------|----------|
| 整理成本 | 记录后需要手动分类、打标签、建链接 | “记完就忘，整理太累” |
| 长期维护成本 | tag 越来越乱、文件夹越来越杂 | “知识库越做越难用” |
| 知识激活问题 | 内容沉没、长期未复用、找不到 | “记了很多，但用不上” |
| 表达启动成本 | 想法还很模糊时，很难直接写成结构化条目 | “脑子里有东西，但不知道怎么开始记” |
| 工具迁移成本 | Notion → Obsidian 搬运痛苦 | “换工具等于重新开始” |

### 3.2 Atlax 解决的核心命题

Atlax 不是“更好的笔记软件”，而是：

**一个降低整理成本、降低表达启动成本、让知识沉淀更轻松的工作台。**

核心价值：
- 不强迫用户先学会管理，才能开始积累知识
- 通过 Classic Mode 让主动整理者获得清晰生产力
- 通过 Chat Mode 帮助懒整理用户或表达启动困难用户先开口、再逐步成形
- 通过 Tag 体系降低分类成本
- 通过可信的推荐能力给出可解释、可修正的建议
- 通过结构化视图让内容可管理、可回看

## 5. 双模式交互系统

### 5.1 系统定义

Atlax 采用双模式交互系统：Classic Mode 与 Chat Mode。

两种模式不是两个独立产品，也不是两个独立功能系统，而是同一知识库的两种交互入口。区别在于用户如何进入知识整理流程，而不在于系统背后维护两套能力边界。

### 5.2 Classic Mode 定义

Classic Mode 面向生产力与显式操作，负责：
- 结构浏览与清晰导航
- Dock 列表查看与待整理内容管理
- Tag 选择、归档、二次编辑、基础数据库视图与筛选
- Archive、Browse、Re-organize 等明确操作
- 适合愿意主动管理知识结构、希望高效率处理内容的用户

### 5.3 Chat Mode 定义

Chat Mode 面向低门槛表达、懒整理用户与表达启动困难用户，负责：
- 帮助用户在想法尚未成形时先开始表达
- 通过 Agent 主动提问、追问、补全上下文，把模糊内容逐步整理为结构化条目
- 在输入过程中协助确认 Tag、类型、归属与后续归档动作
- 用更低心理负担的方式把“先说一点”推进到“可以沉淀”

### 5.4 两种模式的关系

Classic Mode 与 Chat Mode 必须满足以下原则：
- 共用同一知识库
- 共用同一主流程
- 共用同一 Tag 体系
- 共用同一归档与回顾机制
- 不做两套对象模型、不做两套内容系统、不做两套归档逻辑

模式切换只改变展示与交互方式，不改变底层知识数据，不造成内容割裂。

### 5.5 双模式产品原则

| 原则 | 说明 |
|------|------|
| 同一知识库，两种交互方式 | Classic / Chat 只是入口差异，不是能力边界差异 |
| 经典模式提供生产力 | 面向显式操作、结构整理、清晰浏览 |
| 聊天模式提供表达启动与引导 | 面向低门槛记录、主动追问、渐进整理 |
| 主流程统一 | 无论从哪种模式开始，最终都要汇入 Dock → Suggest / Tag → Archive → Browse |
| 模式切换是核心体验 | 模式切换要被感知为思维状态切换，而不是普通 tab 切页 |

## 6. 核心主线功能

### 6.1 主线流程

```text
Classic Capture / Chat Capture
  → Dock
  → Suggest / Tag Confirmation
  → Archive
  → Browse
  → Re-organize
```

### 6.2 双入口，单主线

Classic Mode 进入主线的方式：
- 用户通过 Quick Input Bar、展开编辑器、Dock 列表、Tag / Archive / Browse 入口进行显式操作
- 用户可直接新建内容、编辑内容、选择 Tag、确认归档位置
- 适合对结构已经有明确判断的场景

Chat Mode 进入主线的方式：
- 用户通过底部输入区进入对话式记录
- Agent 主动引导用户表达、追问缺失上下文、帮助确认 Tag / 类型 / 归属
- 对话产出的内容不能停留在聊天记录中，必须进入 Dock 并汇入同一整理主线
- 适合想法模糊、表达启动困难、暂时不想主动整理的场景

### 6.3 主线功能定义

| 功能 | 目标 | 说明 |
|------|------|------|
| **Classic Capture** | 显式输入内容 | 文本输入、Quick Input Bar、展开编辑器、不强制立即分类 |
| **Chat Capture** | 引导式输入内容 | Agent 通过提问、追问、确认帮助用户完成低门槛表达 |
| **Dock** | 查看待整理内容 | 列表展示、状态流转（pending → suggested → archived），是统一待整理入口 |
| **Suggest / Tag Confirmation** | 系统建议与用户确认 | 规则引擎、关键词匹配、历史行为信号、Agent 对话确认、可解释 |
| **Archive** | 归档到结构化单元 | Entry 创建、Tag 关联、项目归属；归档后可再次打开并回到 Dock 再整理 |
| **Browse** | 回看与筛选 | Entries 列表、基础筛选、多维度浏览、关系查看入口 |
| **Re-organize** | 二次整理 | 已归档内容可再次编辑、补 Tag、调归属、重新入 Dock |

### 6.4 归档不是一次性终局

- 已归档内容必须支持再次编辑、补充 Tag、调整归属、重新整理。
- Browse 不是只读结果页，而是持续整理流程中的回看与再组织入口。
- Chat Mode 产生的内容也必须具备与 Classic Mode 相同的二次整理能力。
- 产品不能把用户锁死在“一次性纳入”流程里，否则无法支撑真实知识沉淀。

## 7. Chat Mode 产品目标

### 7.1 目标定义

Chat Mode 不是普通聊天窗口，也不是泛化 AI 陪伴入口。它是面向知识工作流的引导式对话能力，其目标是把用户难以立即整理的内容，转化为可沉淀、可归类、可回看的知识条目。

### 7.2 核心能力要求

Chat Mode 至少需要具备以下产品目标：
- Agent 主动引导用户记录，而不是只等待用户提出清晰问题
- Agent 主动追问缺失上下文，帮助补足时间、主题、归属、下一步等关键信息
- Agent 主动帮助用户确认 Tag、内容类型、项目归属与归档动作
- Agent 主动发起回顾与整理提醒，而不是只停留在输入阶段
- Agent 帮助用户从“说不出来”走向“先说一点”，降低启动门槛
- Agent 可以提供适度情绪承接与表达鼓励，但最终目标仍是知识管理与内容沉淀

### 7.3 边界要求

- Chat Mode 不以开放式问答准确率为主要评价标准，而以是否帮助用户完成知识整理主线为主要评价标准。
- Chat Mode 不应产出脱离知识库的孤立对话记录。
- Chat Mode 不应把用户长期留在聊天态，而应适时推动进入 Tag 确认、归档、回看与再组织。
- Chat Mode 不应形成第二套“聊天知识库”。

## 8. Chat Mode 日常使用场景

### 8.1 场景定位

Chat Mode 的价值不在于“可聊天”，而在于把日常知识管理中最难启动、最难坚持、最容易半途放弃的环节变得可持续。以下场景用于指导产品设计、提示词策略与交互优先级。

### 8.2 场景 A：启动表达

适用时机：用户打开产品，但不知道该记什么，或者知道脑子里有内容却难以开头。

典型路径：
- Agent 主动发起一个足够具体但不沉重的问题。
- 用户只需回答一句话或一小段内容。
- Agent 根据回答继续追问，帮助内容成形。
- 最终进入 Dock，等待 Tag 确认与归档。

典型引导示例：
- “今天最值得记录的一件事是什么？”
- “最近哪个项目最卡？”
- “你脑子里最近反复出现什么想法？”

产品要求：
- 提问要帮助用户开口，而不是制造答题压力。
- 第一轮输入门槛必须足够低。
- 后续追问要围绕知识整理，不跑向泛话题陪聊。

### 8.3 场景 B：知识回顾

适用时机：知识库已积累一定内容，系统需要帮助用户重新连接过去记录，形成再次使用动机。

典型路径：
- Agent 基于已有知识库状态发起提醒或追问。
- 用户在提醒下补充进展、修正状态或重新整理旧内容。
- 相关内容被再次归档或更新，避免沉没。

典型引导示例：
- “你已经 7 天没更新项目 X，要不要补一下最近进展？”
- “最近连续记录了焦虑主题，要不要整理一下原因？”
- “上周提到的计划，现在进展如何？”

产品要求：
- 回顾必须基于真实已有内容，不做空泛提醒。
- 提醒目标是促成更新、归档或再整理，不是单纯刷存在感。
- 回顾频率要可控，避免造成打扰。

### 8.4 场景 C：整理推动

适用时机：用户已经输入碎片内容，但尚未完成类型判断、Tag 确认或归档。

典型路径：
- 用户输入一句碎片化内容。
- Agent 判断当前缺的不是“更多内容”，而是“整理决策”。
- Agent 发起整理相关问题，推动内容从 pending 进入 archived。

典型引导示例：
- “这条更像项目、灵感还是待办？”
- “需要归到哪个 Tag？”
- “要不要整理成正式条目？”

产品要求：
- 提问优先帮助用户做最小决策，不一次性问太多。
- 若用户暂时不确定，允许先以待确认状态进入 Dock。
- 整理推动应与 Suggest / Tag 机制衔接，而不是另起一套逻辑。

### 8.5 场景 D：低负担陪伴式记录

适用时机：用户只愿意说一句话，或者处在疲惫、焦虑、表达阻滞状态下。

典型路径：
- 用户只输入一句短语或情绪化表达。
- Agent 不要求用户一次说完整，而是通过非常轻的追问帮助补足最基本上下文。
- 内容先被保住，再逐步进入整理链路。

产品要求：
- 陪伴感是降低表达阻力的手段，不是产品目标本身。
- 对话要持续把内容拉回知识沉淀，而不是停在情绪对谈。
- 即使用户只说一句话，也要能进入同一知识库并在之后继续整理。

## 9. 用户留存机制

### 9.1 目标定义

Atlax 的留存不依赖强提醒或内容消费，而依赖用户持续觉得“这个系统帮我更轻松地记住、整理并找回重要内容”。

### 9.2 日留存理由

用户第二天回来的原因应包括：
- Agent 每天能发起一次高质量、低负担的记录引导，帮助用户快速开口。
- 输入门槛足够低，想到就能记，不需要先整理清楚。
- 用户能快速找到昨天记录的内容，并继续补充或归档。
- Dock 中仍有待整理内容，形成明确的下一步动作。

### 9.3 周留存理由

用户每周持续回来的原因应包括：
- Weekly Review 基础版能帮助用户看到本周记录概况。
- 本周反复出现的主题、Tag 或项目变化是可见的。
- 项目或主题的进展能被持续追踪，而不是写完就沉没。
- 旧内容能够被重新打开并继续整理，形成回看价值。

### 9.4 长期留存理由

用户长期留下来的原因应包括：
- 知识库越来越有结构，而不是越来越乱。
- 过去内容找得到、接得上、用得起来。
- 用户能明显感知系统越来越理解自己的 Tag、主题与整理习惯。
- Atlax 开始替代备忘录、微信收藏夹和零散文档，成为默认记录入口。

### 9.5 留存机制的产品约束

- 留存必须建立在真实记录与整理价值上，不能依赖空洞提醒。
- Chat Mode 的主动引导是留存抓手，但必须服务于知识管理主线。
- Weekly Review、回顾提醒、Dock 待整理状态应形成闭环，而不是彼此割裂。

## 10. Tag 正式能力

### 10.1 Tag 定位

Tag 是 Atlax 当前阶段最重要的结构骨架之一，不只是筛选器，也不是可有可无的小点缀。

| 定位 | 说明 |
|------|------|
| 分类服务 | Tag 服务内容分类，降低整理成本 |
| 结构骨架 | Tag 是当前知识结构感的核心接口之一，支撑主题聚合、Entry 汇拢与树状浏览感 |
| 检索服务 | Tag 服务后续检索、聚合 |
| 导入映射 | Tag 服务未来导入能力的数据映射 |
| 关系可视化 | Tag 服务关系可视化、关联发现；默认以清晰视图与推荐为主，不直接暴露高学习成本的复杂编辑器 |

### 10.2 Tag 与 Suggestion 并存原则

```text
用户主动选择 Tag → 始终优先按用户意图归类
用户不选 Tag → 系统给出低打扰建议，而不是强制替用户决定
用户接受建议 → 按建议归类，并记录为后续偏好信号
用户修正建议 → 以用户修正结果为准，并用于优化后续建议
聊天模式中 Tag 未确定 → Agent 通过对话引导用户确认或先给出待确认建议
```

### 10.3 可信推荐原则

- 推荐质量直接影响用户对产品的信任，错误建议不能长期重复出现。
- 推荐优先依赖本地算法与本地信号，避免默认上传原始内容。
- 推荐必须可解释，至少要让用户理解“为什么会推荐这个 Tag / 关系”。
- 推荐必须可关闭、可忽略、可修正，不能把错误建议变成新的使用负担。
- 在 Chat Mode 中，推荐不仅体现在 UI 标签建议上，也体现在 Agent 的追问与确认策略中。

### 10.4 Tag 是当前不接 LLM 时的关键减压设计

- Tag 让用户可以主动控制分类。
- Tag 规则引擎可以给出可解释的建议。
- Tag 不依赖大模型才能成立。
- 在不强依赖 LLM 的阶段，Tag 是组织效率与可维护性的关键。
- Tag 同时支撑未来导入映射、主题聚合与树状浏览感构建。

## 11. Demo 界面目标

### 11.1 设计原则

| 原则 | 说明 |
|------|------|
| 必须像真正的生产力工具 | 不是只有输入框的原型 |
| 尽量单页完成主要操作 | 减少频繁路由跳转 |
| 首页必须极简 | 不堆复杂卡片，不堆大量多页面入口，更像安静的启动面板 |
| 输入能力必须覆盖长短内容 | 不能只提供一个适合短句的记录框，长笔记 / 长文章也必须有合理入口 |
| 输入区不能破坏主工作台结构 | 输入组件不应横跨两个模块，避免版式割裂与阅读层级混乱 |
| 模式切换是核心体验的一部分 | 需要有状态感、轻动画与明显心智转换 |

### 11.2 主界面结构

| 区域 | 功能 |
|------|------|
| Sidebar | 导航、模式切换、对象列表、Tag / 项目 / 视图入口 |
| Workspace | 内容预览、整理面板、详情展示、Browse 主内容区 |
| Input Bar | 统一输入入口，可承载经典快速输入或聊天起点 |
| Expanded Editor | 面向长笔记 / 长文章 / 富文本输入的大编辑器，可由输入区展开进入 |
| Action Area | 统一操作区（归档、删除、编辑、确认建议等） |

### 11.3 首页原则

- 首页保持极简，只承担进入工作状态的职责。
- 首页优先提供模式进入、继续整理入口、最近内容入口三类必要信息。
- 首页不堆复杂信息卡片，不做多页面导航分发。

### 11.4 模式切换体验

- 模式切换是产品核心识别点之一。
- 模式切换应更像滑动开关或状态控制器，而不是普通 tab。
- 切换时页面内容需要有轻动画过渡，强调“进入不同思维模式”。
- 切换只影响展示与交互方式，不影响底层知识数据。

### 11.5 Capture 输入交互原则

- Capture 不能只有单一的小输入框。产品既要支持灵感、短句、待办，也要支持较长笔记、整篇文章与高信息密度记录。
- 底部 Input Bar 是统一输入入口，但在 Classic / Chat 下呈现不同交互状态。
- 快速输入区应提供明确的扩展动作；用户进入展开态后，应看到更适合长内容输入的编辑体验。
- 快速输入区与展开编辑器应属于同一条 Capture 主线，而不是两个割裂功能。
- Phase 2 先解决“能不能合理输入长短内容”、“双模式是否能共用输入主线”和“布局是否成立”；Phase 3 再处理富文本细节与精细交互。

### 11.6 UI / 交互规范真源

- `docs/product/FRONT_DESIGN.md` 是 Atlax 前端视觉、交互、动效规范唯一真源。
- 自 Phase 2 起，所有前端功能新增与改造必须遵循该规范（包括色彩、层次、模式切换、输入渐进展开、动效曲线、Light/Dark 一致性）。
- 不允许以“改版”名义删除既有主线能力；必须在统一设计语言下补齐和保持功能闭环。

## 13. 当前非目标与延后能力

### 13.1 当前非目标

| 非目标 | 说明 |
|------|------|
| 重型任务管理器 | 当前不做复杂任务系统或 GTD 系统 |
| 复杂协作系统 | 当前不做多人实时协作与细粒度权限系统 |

### 13.2 延后但保留方向

| 延后项 | 当前处理方式 |
|------|------|
| 完整导入能力 | Phase 4 再推进，当前只留入口与接口 |
| 搜索增强与关系增强 | Phase 4 再做，当前保留基础浏览与结构约束 |
| 语音输入等扩展采集 | Phase 4 再评估 |
| 同步与多端扩展 | Phase 5 再推进，当前不否定未来同步 |
| AI 增强建议 | Phase 5 再加强，当前以规则与引导式交互为主 |
| 多人协作 | Phase 5 再评估 |

### 13.3 必须预留的基础原则

- 账号系统不是可选项，Demo 阶段就需要登录后管理个人身份与工作区。
- 本地优先不等于服务端持有全部内容；隐私策略应以本地处理、最小上传、用户授权同步为原则。
- 同步不是当前主交付，但产品原则上不能否定未来同步；后续优先考虑用户已有云盘、系统云能力或用户授权外部存储。

## 16. 数据结构原则

### 16.1 树状感是用户体验目标，不是底层存储约束

| 层级 | 说明 |
|------|------|
| 用户角色 | 用户是 root，是知识组织的起点 |
| 组织骨架 | Tag 提供轻量组织骨架 |
| 内容聚合 | Notes / Ideas / Projects / Reviews 可围绕 Tag 聚合 |
| 用户体验层 | 可以提供“像树一样有层次感”的组织体验 |
| 底层数据模型 | 采用适合查询、关联、过滤、扩展的结构化模型 |
| 不要求 | 不要求真实物理存储为树，也不要求强制手工维护复杂双链 |

### 16.2 数据模型原则

| 原则 | 说明 |
|------|------|
| 不绑定 UI 视图 | Domain model 不绑定 UI 视图 |
| 适合查询 | 不能使用只适合展示、不适合查询的数据结构 |
| 预留扩展 | 为后续筛选、关系查询、Review 聚合留空间 |
| 同一知识库 | Classic / Chat / Import 都必须写入同一对象模型 |
| 结构自然长出 | 产品目标不是要求用户手工搭建复杂双链，而是通过 Tag、Entry 聚合、项目挂载、关系浏览让结构自然长出来 |

## 17. 产品验证指标（North Star Metrics）

### 17.1 指标目标

Atlax 的指标设计用于回答四个核心问题：
- 用户是否持续记录
- 用户是否觉得整理更轻松
- Chat Mode 是否真的被使用并推动归档
- 产品是否开始替代备忘录、微信收藏夹和零散记录方式

### 17.2 核心指标

| 指标 | 定义 | 目的 |
|------|------|------|
| DAU | 每日活跃用户数 | 判断产品是否形成日常使用习惯 |
| 每用户日均记录次数 | 每个活跃用户每天产生的记录次数 | 判断产品是否真的承接了日常输入 |
| Chat 引导后完成归档率 | 经 Chat 引导后最终进入 Archive 的内容占比 | 判断 Chat Mode 是否真正服务知识整理 |
| 7 日留存率 | 新用户 7 天后是否仍然回来使用 | 判断产品是否具备持续价值 |
| Weekly Review 打开率 | 有资格用户中打开 Weekly Review 的比例 | 判断回顾机制是否形成周留存理由 |

### 17.3 产品健康指标

| 指标 | 定义 | 目的 |
|------|------|------|
| 平均每用户知识条目数 | 每位用户累计归档条目数 | 判断知识库是否在持续沉淀 |
| Tag 使用率 | 记录中带有 Tag 的内容占比 | 判断结构化能力是否真实发生 |
| 搜索使用率 | 使用搜索或浏览回找内容的用户占比 | 判断知识是否开始被回用 |
| Chat Mode 使用占比 | 活跃用户中使用 Chat Mode 的比例 | 判断双模式是否成立，而非只有 Classic 在工作 |

### 17.4 初期验证目标

上线初期至少应验证以下结果：
- 用户能连续多天产生记录，而不是一次性试用。
- 用户能在 Chat Mode 中完成从表达启动到归档的闭环。
- 用户能在第二天快速找到昨天记录的内容并继续整理。
- 用户开始用 Atlax 替代至少一种零散记录方式。
- Tag、Dock、Archive 没有成为额外负担，而是提升了整理完成率。

| Archive | 用户能完成归档，内容进入 Entries，且可重新打开继续整理 |
| Browse | 用户能找到归档内容，完成查看、打开与再次整理 |
| Same Knowledge Base | 模式切换后仍作用于同一知识库，同一条内容可在两种模式下继续整理 |
| Chat Product Boundary | 聊天模式体现为知识整理引导，而不是普通 AI 问答框 |
| Login / Workspace | 用户必须能登录并进入自己的工作区，身份状态在刷新后可保持 |
| UI | 像真正的生产力工具，不是只有输入框的原型；主工作台排版边界清晰 |

### 18.2 产品验收

Phase 2 Demo 通过的判断：
- 用户能快速记录，不被结构打断。
- 用户能在经典模式下完成一次整理闭环。
- 用户能在聊天模式下完成一次被引导式输入与归档。
- Tag / Suggest / Archive 在两种模式下都成立。
- 用户能明确感知产品不是两套系统，而是同一知识库的两种交互方式。
- 这个版本看起来真实能上线、能被使用、能验证，而不是功能堆叠后的展示品。
