## 1. MVP 目标

Atlax 当前 MVP 的唯一目标不是“做出一个完整知识管理平台”，而是验证下面这个闭环是否成立：

> **用户输入任何内容后，系统能先接住，再帮用户完成初步整理，并在回顾场景里让用户感受到“知识在替我工作”。**

### 1.1 MVP 要验证的三件事

| 验证项 | 说明 | 是否为 MVP 必须验证 |
|------|------|----------------|
| 输入自由是否成立 | 用户能否不做预分类就快速记录 | 是 |
| 系统代整理是否成立 | 系统建议是否足够让用户愿意一键确认 | 是 |
| 回顾激活是否成立 | 用户是否会在 Review 场景里重新看到价值 | 是 |

### 1.2 MVP 的真实交付形态

Atlax 当前 MVP 不是“全平台知识操作系统”，而是一个**Web Demo / Web-first 单人产品原型**：

- 单用户使用
- 本地优先
- 浏览器本地存储为主
- 默认不依赖独立后端服务
- 默认不依赖云端数据库
- 默认不依赖 LLM 才能完成核心闭环

这意味着：

- **MVP 可以演示真实工作流**，但不承诺多设备同步
- **MVP 可以导出 Markdown**，但不承诺浏览器里直接管理真实本地 Markdown 工作区
- **MVP 可以提供“AI 整理”体验**，但底层优先用规则引擎与轻量启发式实现

## 2. MVP 范围总览

### 2.1 P0：必须完成的 4 个核心模块

| 模块 | 目标 | MVP 交付内容 | 技术落地方式 |
|------|------|-------------|-------------|
| **Capture** | 快速把内容丢进系统 | 文本输入、快捷新建、基础语音转文字入口 | 表单输入 + 浏览器语音能力（支持时启用） |
| **Inbox AI 整理** | 系统先给建议，用户一键确认 | 类型建议、标签建议、项目建议、归档操作 | 规则引擎 + 关键词匹配 + 最近项目召回 |
| **Database View** | 让用户能像数据库一样管理内容 | Entries / Projects / Tasks 的 Table/List 视图 | 本地结构化存储 + 表格组件 |
| **Weekly Review** | 让用户感受到知识被激活 | 本周新增、本周整理率、项目进展、健康摘要、提醒卡片 | 聚合统计 + 启发式规则 |

### 2.2 P1：紧接 MVP 的增强项

这些能力与 PRD 保持一致，但**不作为第一轮必须交付**，而是建立在 P0 闭环跑通之后再加：

| 模块 | 定位 | P1 的务实落地方式 |
|------|------|------------------|
| **Knowledge Health** | 对知识库状态做体检 | 先用规则计算分数与问题清单，不做复杂图谱 |
| **Insight Engine** | 主动提醒用户下一步 | 先做规则触发的提醒卡片，不做 Agent |
| **Markdown Export** | 数据可迁移 | 单条/批量导出为 Markdown + frontmatter |
| **Search** | 快速复用内容 | 标题、正文、标签搜索 |
| **Links 建议** | 轻量关联发现 | 基于标签/关键词相似度给出建议，不自动写入 |
| **Settings** | 基础配置 | 导出设置、提醒开关、语音能力开关 |
| **Editor** | 内容编辑 | 简单 Markdown 文本编辑，不追求重型编辑器能力 |

### 2.3 当前阶段明确不做

| 不做项 | 原因 |
|------|------|
| 多人协作 | 会引入账户、权限、云端存储，偏离 MVP |
| 实时同步 | 需要远程状态协调，技术复杂度过高 |
| Notion 全量导入 | 数据映射复杂，不适合第一阶段 |
| 本地 Markdown 工作区双向同步 | 浏览器环境落地成本高，且容易产生一致性问题 |
| 高级 Graph / 知识地图 | 展示价值大于实际核心价值，先后置 |
| AI 问答 / AI Agent | 会显著提升成本与复杂度，且不直接验证核心命题 |
| 手写输入 | 采集、识别、数据结构都较重 |
| Billing / 企业版 / 权限系统 | 当前不是商业化验证阶段 |

## 3. 核心模块定义

### 3.1 Capture（P0）

**目标**：让用户在最低认知负担下完成记录。

**MVP 交付**：
- 文本输入
- 无标题保存
- 不强制分类
- 新建后默认进入 Inbox
- 浏览器支持时提供语音转文字入口

**边界约束**：
- 不做离线语音识别
- 不做长音频文件管理
- 语音输入失败时，允许回退为手动文本整理

### 3.2 Inbox AI 整理（P0）

**目标**：让“系统代整理”成为用户第一次感知到的价值。

**MVP 交付**：
- Inbox 列表
- 单条内容建议面板
- 建议字段：`type`、`tags`、`project`
- 一键接受建议
- 手动微调后归档
- 批量处理（可选，若排期不足可延后到 P1）

**技术原则**：
- “AI 整理”是产品表达，不等于必须调用 LLM
- 默认使用规则引擎、关键词词典、最近上下文匹配
- 建议永远是建议，不做自动强制归档

### 3.3 Database View（P0）

**目标**：让内容在进入系统后变得可管理、可筛选、可复用。

**MVP 交付**：
- `Entries` 视图：查看所有笔记/记录
- `Projects` 视图：按项目聚合内容
- `Tasks` 视图：展示从内容中抽取或手动创建的任务
- 支持 Table / List 两种基本视图
- 支持按 `type`、`status`、`project`、`tag` 过滤

**边界约束**：
- 不做 Board / Timeline / Calendar
- 不做复杂自定义数据库字段系统
- 不做多人共享数据库

### 3.4 Weekly Review（P0）

**目标**：让用户看到“本周我输入了什么、整理了什么、哪些内容值得回看”。

**MVP 交付**：
- 本周新增内容数
- 本周 Inbox 处理率
- 本周项目活动摘要
- 健康摘要（沉没内容 / 孤岛内容 / 停滞项目）
- 提醒卡片（例如项目停滞、主题值得汇总）
- 可分享的简版成长报告（图片卡片可作为后续增强）

**务实说明**：
- Weekly Review 是 MVP 的关键价值展示页
- `Knowledge Health` 与 `Insight Engine` 在第一阶段优先作为 Review 内能力呈现，而不是先拆成两个复杂独立系统

## 4. Knowledge Health 与 Insight Engine 的 MVP 定义

### 4.1 Knowledge Health（P1，但在 Review 中有基础形态）

在 PRD 中这是产品差异点，但技术上第一阶段不需要做成复杂评分平台。

**MVP / P1 规则版能力**：
- 长时间未打开的归档内容
- 没有关联项目的高频主题
- 14 天无更新的活跃项目
- 没有标签且未归档的长期 Inbox 项

**不做**：
- 复杂图神经网络或向量知识图谱
- 自动修复
- 大规模历史趋势分析

### 4.2 Insight Engine（P1，但在 Review 中有基础形态）

Insight Engine 在 MVP 里定义为：

> **基于规则生成的下一步提醒系统，而不是问答机器人，也不是自治 Agent。**

**MVP / P1 规则版提醒示例**：
- 某项目 10 天未更新，建议回顾卡点
- 同主题内容连续出现 5 次，建议建立项目页
- 某笔记 30 天未再打开，建议回顾

**不做**：
- 自动执行行动
- 主动外发消息推送
- 多轮对话式智能助手

## 5. Demo 最小闭环

### 5.1 演示路径

1. 用户输入一段文本或一条语音转写内容
2. 系统将内容写入 `Inbox`
3. 系统生成类型 / 标签 / 项目建议
4. 用户一键确认并归档
5. 内容出现在 `Database View`
6. 用户打开 `Weekly Review`
7. 看到本周新增、整理率、项目摘要
8. 看到一条规则生成的 Insight 提醒
9. 导出一条 Markdown，确认数据可带走

### 5.2 演示文案

**首页核心文案保持与 PRD 一致**：

> **记录一切，剩下交给 Atlax。**

## 6. 验收标准

### 6.1 功能验收

| 项目 | 验收标准 |
|------|----------|
| Capture | 用户可在 10 秒内完成新建内容并进入 Inbox |
| Inbox | 每条内容都能看到至少一组结构化建议 |
| Archive | 用户可以接受建议后成功归档 |
| Database | 用户可以在列表中重新找到归档内容 |
| Review | 用户能看到本周统计和至少一条提醒卡片 |
| Export | 用户可以导出单条 Markdown 文件 |

### 6.2 产品验收

MVP 通过的判断不是“功能齐不齐”，而是是否满足下面三点：

- 用户能快速记录，不被结构打断
- 用户会觉得整理比自己手动做更轻松
- 用户在 Review 页能明显感受到系统提供了额外价值

## 7. 里程碑建议

| 阶段 | 目标 | 交付重点 |
|------|------|---------|
| Week 1 | 跑通本地数据与 Capture | 录入、存储、Inbox 列表 |
| Week 2 | 跑通建议与归档 | Suggestion Engine、归档流程、基础视图 |
| Week 3 | 跑通 Database 与 Review | 统计、项目摘要、健康摘要 |
| Week 4 | 收尾与 Demo 打磨 | 导出、体验优化、示例数据、演示脚本 |

---

## 8. 范围判断原则

如果一个需求不直接增强以下闭环，就不进入当前 MVP：

**输入 → 进入 Inbox → 接受建议 → 进入 Database → 在 Review 中被重新激活**

这条原则用于防止 MVP 再次膨胀成“另一个什么都想做的知识平台”。
