## Atlax 第一版独立后端（P0 闭环）实施计划

### Summary
基于你已确认的约束，第一版后端按 `NestJS + Prisma + SQLite` 落地独立服务，只交付 MVP P0 主闭环能力：`Capture → Inbox Suggest → Archive → Database 查询 → Weekly Review`。  
规范优先级以 `PRD + TECH_SPEC + ARCHITECTURE` 为准，长文档仅作补充；严格不做当前阶段未要求能力。

### Implementation Changes
1. 工程与分层
- 在 `apps/server` 搭建 NestJS 服务骨架，按模块拆分：`capture`、`inbox`、`archive`、`entries`、`projects`、`tasks`、`review`、`suggestion`、`health`、`insight`。
- 持久层使用 Prisma + SQLite；服务间通过明确 DTO/Service 调用，不共享可变状态。
- 统一响应协议：`{ code, message, data }`。

2. 数据模型（P0 必需）
- `InboxEntry`: `id/rawText/sourceType/status(created enum: pending|suggested|archived)/createdAt/suggestedAt/archivedAt`。
- `Entry`: `id/title/content/type/status/tags/projectId/createdAt/updatedAt/viewCount/lastViewedAt`。
- `Project`: `id/name/status/description?/lastActivityAt/createdAt/updatedAt`。
- `Task`: `id/title/status/projectId/sourceEntryId/dueAt?/priority?/createdAt/updatedAt`。
- `InsightCard` 与 `WeeklyReviewSnapshot` 作为 Review 聚合输出结构（可按需落库，首版默认实时计算）。
- 迁移策略：先建最小可运行 schema，再补充索引（`createdAt/status/projectId/type`）。

3. API（第一版可运行契约）
- `POST /api/capture`：创建 Inbox 条目。
- `GET /api/inbox`：获取待整理列表。
- `POST /api/inbox/:id/suggest`：规则引擎生成 `type/tags/project` 建议。
- `POST /api/inbox/:id/archive`：接受/微调建议并归档；同步创建或更新 Project/Task。
- `GET /api/entries`：Entries 列表与过滤（`type/status/project/tag`）。
- `GET /api/projects`：Projects 列表（Database View 必需读接口）。
- `GET /api/tasks`：Tasks 列表（Database View 必需读接口）。
- `GET /api/review/weekly`：周统计 + Health 信号 + Insight 卡片。

4. 规则引擎（纯规则，无 LLM）
- 类型识别：关键词规则 + 默认回退 `note`。
- 标签建议：关键词词典 + 历史高频标签召回。
- 项目建议：最近活跃项目优先 + 名称关键词匹配。
- 任务抽取：`TODO/待办/下周要做` 等模式匹配生成 Task。
- Health 规则：`unused_entries(30d)`、`stalled_projects(14d)`、`orphan_entries`、`unstructured_inbox`。
- Insight 规则：`project_stalled`、`topic_cluster`、`knowledge_activation` 三类提醒。

5. 非目标（本阶段明确不做）
- 鉴权/账户体系、多用户协作、实时同步、云端存储。
- Import/Export/Search/AI 调用链路。
- 复杂 Agent、复杂 schema 自定义、权限系统。

### Test Plan
- 单元测试
1. `suggestion`：类型/标签/项目建议命中与回退。
2. `archive`：状态流转 `pending→suggested→archived`、Entry/Project/Task 写入一致性。
3. `review/health/insight`：周聚合、健康信号、提醒卡片生成规则。
- 端到端测试
1. 主链路：`capture -> suggest -> archive -> entries/projects/tasks -> weekly review`。
2. 异常链路：空输入、非法状态归档、不存在 ID、重复归档防护。
3. 契约检查：所有接口保持 `code/message/data` 结构与正确 HTTP 状态码。
- 验收对齐
1. 10 秒内可完成“输入并进入 Inbox”。
2. 每条 Inbox 至少返回一组结构化建议。
3. 归档后可在 Database 三对象视图中检索到结果。
4. Weekly Review 返回周统计且至少可生成一类提醒。

### Assumptions
- 单用户、单工作区、无鉴权是第一版默认前提。
- 数据库存储 UTC ISO 时间；前端自行做时区展示。
- 文档冲突时按 `PRD + TECH_SPEC + ARCHITECTURE` 解释执行。
- 第一版目标是“可运行闭环后端”，不是“完整产品后端平台”。
