## 一、阶段定位

Phase 2 不只是把 Atlax 做成能演示的脚手架，而是要建立一套真正可用的 Demo 基线。

这一阶段必须完成主线闭环、账号闭环、知识库浏览与再次整理闭环，同时修正当前前端里最影响可用性的两个问题：

- 输入能力过窄，只适合短句，不适合长笔记 / 长文章
- 输入区版式破坏主工作台结构，看起来像横跨两个模块的临时表单

Phase 2 的目标不是把前端打磨到最终形态，而是先把输入能力与布局结构做正确，让 Demo 至少具备可被真实试用的产品骨架。

---

## 二、Phase 2 阶段目标

### 目标 1：完成主线功能闭环

基于当前 PRD，Phase 2 必须至少跑通以下主线能力：

- Capture：快速输入内容进入系统
- Dock：查看待整理内容与当前状态
- Suggest / Tag：系统给建议，用户确认或修正
- Archive / Organize：完成结构化归档
- Browse / Review：找到已归档内容，并具备基础回看入口
- Re-organize：已归档内容可重新打开继续整理

### 目标 2：账号系统在 Demo 阶段成立

账号系统不能后置到后续阶段。

Phase 2 必须让用户能够进入自己的工作区，管理自己的内容身份，并为后续同步与协作授权打基础。

最小要求包括：

- 注册 / 登录 / 退出
- 刷新后身份状态保持
- 用户内容与工作区身份绑定
- 后续同步与授权边界可扩展

### 目标 3：先修正 Capture 输入能力

当前输入方式不能只依赖一个底部小记录框。

Phase 2 需要建立双形态输入：

- 快速输入：负责灵感、短句、短文本、临时记录
- 展开输入：负责长笔记、长文章、结构化记录与更大的编辑空间

这一阶段可以先做到“合理可用”，不要求把富文本细节全部打磨完，但必须保证：

- 用户有明确入口从快速输入切换到展开输入
- 长内容输入不再被小输入框容量限制
- 两种输入形态属于同一条 Capture 主线，而不是分裂成功能孤岛

### 目标 4：把工作台布局纠偏到正确结构

当前输入框横跨模块，会破坏左中右工作台结构的阅读边界。

Phase 2 必须先把输入区放回合理位置，让整体排版成立：

- 输入区不横跨两个主模块
- 列表区、详情区、输入区边界清晰
- 用户能一眼看懂“哪里看列表、哪里看详情、哪里输入内容”

### 目标 5：把 Demo 做成“像产品”而不是“像脚手架”

当前前端不能停留在只有输入框和保存按钮的层级。

Phase 2 需要将界面提升到可演示状态，至少具备以下特征：

- 单页式主工作区
- 左侧导航 / 对象列表
- 右侧内容区 / 建议区 / 预览区
- 常驻快速输入入口 + 展开输入入口
- 不依赖频繁页面跳转完成核心操作
- 整体风格像真正的生产力工具，而不是表单页拼接

### 目标 6：降低“纯推荐整理”带来的信任风险

Phase 2 不能把整理完全压到推荐算法上。

应采用 **Suggestion + Tag 双轨机制**：

- 用户可手动选择 Tag
- 用户不选时，系统提供建议
- 用户明确选择时，以用户意图为准
- 推荐系统作为辅助，而不是唯一入口

### 目标 7：为后续结构增强与同步协作留接口

Phase 2 不要求把图谱、导入、同步、AI 或多人协作完整做完，但必须在数据模型和架构边界上预留位置，包括：

- Tag 体系与 Tag 链路推荐
- Entry 与 Tag / 项目 / 关系的关联
- 导入流程的接入点
- 同步与授权接口
- AI provider 扩展位
- 多人协作的数据与权限边界

## 三、Phase 2 阶段拆分

### Phase 2.1｜统一主工作区、账号入口与版式结构

### 阶段目标

搭建可承载 Demo 主线操作的主界面骨架，并先把版式边界做正确。

### 需要完成的内容

- 建立统一主布局，建议结构为：
    - 左侧：导航 / 工作区入口
    - 中间：当前列表或主内容区
    - 右侧：详情 / 建议 / 预览区
    - 局部快速输入区：短输入入口
    - 展开输入区：长内容编辑入口
- 接入最小账号登录闭环
- 让用户进入自己的工作区，而不是匿名临时态
- 统一核心页面视觉风格
- 修正输入框横跨模块的问题，恢复清晰的工作台边界

### 退出标准 / 子阶段验收标准

- 用户进入产品后，能在一个主工作区里理解主要入口
- 用户可以注册 / 登录 / 退出，并进入自己的工作区
- 刷新后身份状态可保持
- Dock、Browse、Review 的布局风格基本统一
- 输入入口清晰可见，且不再以横跨两个模块的方式破坏布局

---

### Phase 2.2｜Capture 双形态输入与 Dock 链路成立

### 阶段目标

让用户既能快速记录灵感，也能写较长内容，并在 Dock 中看到系统接住内容后的第一步整理动作。

### 需要完成的内容

- 保持快速文本输入为第一入口
- 提供明确的展开动作，例如加号按钮、展开按钮或独立入口
- 建立更大的展开编辑器，支持长笔记 / 长文章输入
- 展开编辑器可以先做“基础可用”，不要求 Phase 2 就把富文本细节打磨完
- 新内容进入 Dock 后默认处于 pending
- Dock 需要可见、可理解、可继续操作，而不是纯数据列表
- 单条内容点击后，右侧可看到详情和建议区域
- 支持从 Dock 进入建议确认流程

### 退出标准 / 子阶段验收标准

- 用户可在 10 秒内完成一条短输入
- 用户也可以输入明显超过短输入框容量的长内容
- 快速输入与展开输入属于同一条 Capture 主线
- 新内容写入 Dock 后可立即看到
- 单条内容的查看与后续整理动作是连贯的

---

### Phase 2.3｜Suggestion + Tag 双轨整理机制

### 阶段目标

把“系统代整理”从单纯推荐，升级为“系统推荐 + 用户显式 Tag 选择”的双轨机制，既保留智能感，也降低错误整理风险。

### 需要完成的内容

#### 1. Suggestion Engine 继续保留规则驱动

- 先不接 LLM
- 继续使用 deterministic 规则
- 至少覆盖典型输入类型与基础 Tag 建议

#### 2. 引入 Tag 作为正式整理维度

- 用户可手动添加或选择 Tag
- 系统可推荐 Tag
- 用户明确选择时，以用户选择为准
- 用户不选择时，可直接接受系统推荐

#### 3. 建议面板需要展示的最小字段

- type 建议
- Tag 建议
- 关系 / 归属建议（若有）
- 可执行动作：
    - 接受并归档
    - 忽略
    - 稍后处理
    - 手动修正

### 退出标准 / 子阶段验收标准

- 典型输入能生成稳定建议
- 用户可看到 Tag 建议
- 用户可手动补 Tag / 改 Tag
- 用户修正结果会作为后续偏好信号
- 建议面板足够清晰，不需要依赖解释文档才能理解

---

### Phase 2.4｜Archive 闭环与二次整理能力

### 阶段目标

完成从 Dock 条目到结构化 Entry 的最小归档闭环，并保证归档不是一次性终局。

### 需要完成的内容

- 接受建议后，将条目结构化归档为 Entry
- 原条目状态完成迁移
- Tag、type、项目归属等字段进入正式存储层
- 归档后内容能在 Browse 中重新找到
- 已归档内容可重新打开并回到整理流程

### 最低状态流要求

至少保证以下状态流成立：

- pending → suggested
- suggested → archived
- suggested → ignored
- archived → re-organize

### 退出标准 / 子阶段验收标准

- 用户接受建议后可成功归档
- 归档动作不会丢失原始内容
- 归档后的内容能在结构化视图中再次找到
- 已归档内容可重新编辑、补 Tag、调整归属
- 单条完整流程可稳定演示：输入 → 建议 → 确认 → 归档 → 再次整理

---

### Phase 2.5｜Browse / Review 最小可用版本

### 阶段目标

让归档后的内容真正可回看、可筛选、可再次整理，而不是“进库即消失”。

### 需要完成的内容

- 至少提供基础 Entries 浏览视图
- 支持最基础筛选：
    - type
    - status
    - Tag
    - 项目归属
- 点击条目后可查看详情与关系入口
- 提供最小 Review 能力与基础统计
- 提供再次整理入口

### 退出标准 / 子阶段验收标准

- 用户可以在 Browse / Review 重新找到已归档内容
- 至少支持按 type / Tag / 项目中的基础维度查看
- Browse / Review 不是静态列表，而是有回看与整理入口
- 数据展示与实际归档结果一致

---

### Phase 2.6｜Demo 演示路径封板与样例数据完善

### 阶段目标

把整个 Demo 变成可稳定演示、可讲清主线、可让别人快速看懂的产品原型。

### 需要完成的内容

- 整理一套标准 Demo 演示路径
- 准备样例数据，避免空库演示
- 核心文案统一
- 核心页面空状态、无数据状态、异常提示补齐
- 对最容易翻车的流程进行手工回归

### 推荐演示路径

1. 用户登录并进入自己的工作区
2. 通过快速输入记录灵感，或展开编辑器写长内容
3. 内容进入 Dock
4. 系统给出 type / Tag / 关系建议
5. 用户接受或微调后归档
6. 内容进入 Browse / Review
7. 用户重新找到该内容，并可再次整理
8. 完成一次“系统接住内容并帮助持续整理”的演示

### 退出标准 / 子阶段验收标准

- Demo 路径可从头到尾跑通
- 不依赖开发者口头解释也能大致看懂
- 空状态、无数据状态不会让演示中断
- 演示数据足够体现产品价值，而不是只有 1 条测试数据

## 四、Phase 2 总验收标准

Phase 2 只有在以下条件同时满足时，才能视为阶段完成。

### 1. 功能闭环验收

必须跑通：

- Capture 短输入
- Capture 长内容输入
- Dock
- Suggestion + Tag
- Archive 入库
- Browse / Review
- 已归档内容再次整理

### 2. 账号与身份验收

必须做到：

- 用户可登录并进入自己的工作区
- 刷新后身份状态保持
- 内容与工作区身份绑定
- 同步与协作授权边界可扩展

### 3. 产品表达验收

必须做到：

- 界面已经像一个生产力工具，而不是 demo 表单
- 用户进入系统后能理解主界面结构
- 关键操作不需要频繁跨页面跳转
- 输入区不横跨两个主模块，主工作台版式边界清晰

### 4. 数据闭环验收

必须保证：

- 输入内容不丢失
- 建议结果能进入结构化存储
- Tag / type / 关系的最小组织成立
- 已归档内容可重新打开继续整理

### 5. Demo 可演示验收

必须具备：

- 一条完整演示路径
- 一组高质量样例数据
- 页面空状态和错误提示不至于让演示崩掉
- 非开发者也能大致理解产品要解决什么问题

## 五、Phase 2 暂时后置 / 不在本阶段强行实现的能力

这些能力不是不要，而是 **Phase 2 先记录、先预留接口、暂不强行做完**。

### 1. 完整导入能力

暂不在 Demo 阶段完整交付。

但必须预留导入入口与 importer interface。

### 2. 搜索增强与复杂关系计算

暂不在 Demo 阶段完整交付。

但必须在数据模型与视图层预留搜索、关系、图谱扩展方向。

### 3. 富文本精细交互打磨

Phase 2 只要求展开编辑器与输入布局达到合理可用。

富文本工具栏层级、细致动画、输入反馈微交互等放入 Phase 3 继续打磨。

### 4. AI 增强建议

暂不依赖 LLM 才成立。

但必须保留本地算法优先、AI provider 后续接入的扩展方向。

### 5. 云同步与多人协作

暂不在 Demo 阶段完整交付。

但必须预留授权、同步、共享、权限扩展位。
