## 一、阶段定位

Phase 2 不是继续零散补功能，而是把 Atlax 从“局部能力可运行”推进到“可以完整演示主线价值的 Demo”。

这一阶段要完成的不是最终产品，而是一个**可被真实理解、可被连续操作、可被完整演示的 Web Demo**。

用户应该能够在一个相对统一的主界面中完成输入、查看 Inbox、接受建议、归档内容、浏览结构化数据，并看到基础 Review 结果，从而真正感受到：

> **记录一切，剩下交给 Atlax。**
> 

Phase 2 的目标不是把所有长期设想一次做完，而是优先交付 PRD / MVP 文档中明确要求的主线能力，并为后续体验打磨、树形知识组织、导入能力、链接能力等留下清晰扩展入口。

---

## 二、Phase 2 阶段目标

### 目标 1：完成 MVP 主线功能闭环

基于现有文档要求，Phase 2 必须至少跑通以下主线能力：

- Capture：快速输入内容进入系统
- Inbox AI 整理：系统生成建议，用户确认或调整
- Database View：归档后内容可被查看、筛选和管理
- Weekly Review：能看到本周输入、整理、项目活动和基础提醒

### 目标 2：把 Demo 做成“像产品”而不是“像脚手架”

当前前端只有输入框和保存，无法承载“知识操作系统”的产品表达。

Phase 2 需要将界面提升到可演示状态，至少具备以下特征：

- 单页式主工作区
- 左侧导航 / 工具栏
- 右侧内容区 / 预览区
- 底部或固定区域输入入口
- 不依赖频繁页面跳转完成核心操作
- 整体风格参考 Codex Desktop / Notion 的简洁生产力界面，而不是表单页拼接

### 目标 3：降低“纯推荐整理”带来的算法压力

当前如果完全依赖规则引擎推荐，系统压力会过早落在 SuggestionEngine 上。

Phase 2 应加入 **tag 作为半自动整理机制**：

- 用户可手动选择 tag
- 用户未选择时，系统提供 tag 建议
- 归档优先尊重用户明确选择
- 推荐系统作为辅助而非唯一入口

这样可以同时满足：

- 降低整理阻力
- 降低算法复杂度
- 保持产品的开放性
- 为未来知识网络和树形结构建立基础

### 目标 4：为后续知识树与导入能力留接口

Phase 2 不要求把树形知识图谱和导入系统完整做完，但必须在数据模型和架构层为未来扩展预留位置，包括：

- tag 体系
- entry 与 project / task / tag 的关联
- review / insight 的聚合来源
- 导入流程的接入点
- links / knowledge graph / tree aggregation 的扩展方向

## 三、Phase 2 阶段拆分

---

## Phase 2.1｜统一主工作区与产品化界面骨架

### 阶段目标

搭建可承载 Demo 主线操作的主界面骨架，让 Atlax 从“页面集合”升级为“像真正工具的工作台”。

### 需要完成的内容

- 建立统一主布局，建议结构为：
    - 左侧：导航 / 工作区入口
    - 中间：当前列表或主内容区
    - 右侧：详情 / 建议 / 预览区（可根据页面复用）
    - 底部或固定输入栏：快速输入
- 核心页面视觉风格统一
- 减少“进入一个页面只能做一件事”的割裂感
- Inbox / Database / Review 至少在信息结构上统一风格
- 首页 / Landing 若暂未重点打磨，至少不影响 Demo 路径

### 建议的 UI 方向

参考原则：

- 不是复刻 Codex Desktop
- 但要学习它的几个关键优点：
    - 左右分区清晰
    - 工作流集中在一个主界面
    - 用户点击即预览，不来回跳转
    - 输入入口始终可达
- 同时吸收 Notion 的优点：
    - 轻、整洁、信息分层自然
    - 数据和内容不是两套完全割裂的界面

### 退出标准 / 子阶段验收标准

- 用户进入产品后，能在一个主工作区里理解主要入口
- 用户不需要频繁页面跳转也能完成主线操作
- Inbox、Database、Review 的布局风格基本统一
- 输入入口清晰可见，且不是隐藏在二级页面里
- Demo 演示时不会给人“只有表单页 + 列表页”的感觉

---

## Phase 2.2｜Capture 与 Inbox 统一输入-待整理链路

### 阶段目标

让用户能把内容快速扔进系统，并在 Inbox 中看到系统接住内容后的第一步整理动作。

### 需要完成的内容

- 保持快速文本输入为第一入口
- 若浏览器能力允许，可保留语音入口，但不作为阻塞项
- 新内容进入 Inbox 后默认处于 pending
- Inbox 需要可见、可理解、可继续操作，而不是纯数据列表
- 单条内容点击后，右侧可看到详情和建议区域
- 支持从 Inbox 进入建议确认流程

### 建议补充细节

- 输入时不强制标题
- 系统可以根据原始文本自动生成临时标题或摘要首行
- 保留 sourceType、createdAt、status 等基础字段
- 对异常输入做最小校验，但不能破坏低摩擦输入体验

### 退出标准 / 子阶段验收标准

- 用户可在 10 秒内完成一条新输入
- 新内容写入 Inbox 后可立即看到
- Inbox 中至少可查看：
    - 原始内容
    - 状态
    - 时间
    - 建议入口 / 建议预览
- 单条内容的查看与后续整理动作是连贯的

---

## Phase 2.3｜Suggestion + Tag 双轨整理机制

### 阶段目标

把“系统代整理”从单纯推荐，升级为“系统推荐 + 用户显式 tag 选择”的双轨机制，既保留智能感，也降低错误整理风险。

### 需要完成的内容

### 1. Suggestion Engine 继续保留规则驱动

- 先不接 LLM
- 继续使用 deterministic 规则
- 至少覆盖：
    - note
    - task
    - meeting
    - idea
    - reading

### 2. 引入 tag 作为正式整理维度

- 用户可手动添加或选择 tag
- 系统可推荐 tag
- 用户明确选择时，以用户选择为准
- 用户不选择时，可直接接受系统推荐

### 3. 建议面板需要展示的最小字段

- type 建议
- tag 建议
- project 建议（若有）
- 可执行动作：
    - 接受并归档
    - 忽略
    - 稍后处理

### 为什么这个设计重要

这一步很关键，因为它解决了两个问题：

第一，纯规则推荐很容易过早变成“算法正确率焦虑”；

第二，用户并不总是想把整理权完全交出去。

加入 tag 后，Atlax 的体验会更像：

- 系统先帮你想一轮
- 你可以快速确认
- 你也可以少量手动纠偏
- 知识组织开始围绕 tag 积累

这比死磕“推荐必须一次猜对”现实得多，也更适合 Demo。

### 退出标准 / 子阶段验收标准

- 至少 5 类典型输入能生成稳定建议
- 用户可看到 tag 建议
- 用户可手动补 tag / 改 tag
- 接受建议后，tag 会进入结构化数据
- 建议面板足够清晰，不需要依赖解释文档才能理解

---

## Phase 2.4｜Archive 闭环与结构化入库

### 阶段目标

完成从 Inbox 条目到结构化 Entry 的最小归档闭环，并让后续 Database / Review 真正有数据可用。

### 需要完成的内容

- 接受建议后，将 InboxEntry 结构化归档为 Entry
- 原条目状态完成迁移
- 支持最小的 project / task 派生能力
- tag、type、project 等字段进入正式存储层
- 归档后内容能在 Database View 中重新找到

### 最低状态流要求

至少保证以下状态流成立：

- pending → suggested
- suggested → archived
- suggested → ignored
- ignored → pending（如支持恢复）

### 建议补充细节

- 对 task 的抽取要保守，不追求“什么都抽”
- project 先保留最小实体，不做复杂层级
- archive 的核心目标是“稳定入库”，不是“高阶知识图谱”

### 退出标准 / 子阶段验收标准

- 用户接受建议后可成功归档
- 归档动作不会丢失原始内容
- 归档后的内容能在结构化视图中再次找到
- 状态迁移可被 UI 正确反映
- 单条完整流程可稳定演示：输入 → 建议 → 确认 → 归档

---

## Phase 2.5｜Database View 最小可用版本

### 阶段目标

让归档后的内容真正可管理，而不是“进库即消失”。

### 需要完成的内容

- 至少提供以下基础视图：
    - Entries
    - Projects
    - Tasks
- 支持 Table 或 List 的最小切换，或者至少一种清晰主视图
- 支持最基础筛选：
    - type
    - status
    - tag
    - project
- 点击条目后可查看详情
- 与 Inbox / 建议面板的交互风格尽量统一

### 这里必须注意的事

Database View 的目标不是做成完整 Notion，也不是马上做出复杂数据库系统。

Phase 2 只需要让用户感觉：

- 内容进来之后不是散的
- 可以再找到
- 可以按结构看
- tag / project / task 开始形成组织感

### 退出标准 / 子阶段验收标准

- 用户可以在 Database View 重新找到已归档内容
- 至少支持按 type / tag / project 中的基础维度查看
- Projects / Tasks 视图不是空壳，至少能展示最小字段
- 数据展示与实际归档结果一致

---

## Phase 2.6｜Weekly Review 基础版落地

### 阶段目标

让用户第一次看到“知识在替我工作”的产品表达，而不是只有输入和整理，没有激活。

### 需要完成的内容

- 本周新增内容数
- 本周 Inbox 处理率
- 本周项目活动摘要
- 至少 1~3 类基础健康信号
- 至少 1 条规则生成 Insight / Reminder 卡片

### 推荐最小健康信号

- 长期未整理 Inbox 项
- 无 tag 无 project 的孤岛内容
- 长时间未更新的项目

### 推荐最小提醒卡片

- 某项目停滞，建议回顾
- 某主题近期频繁出现，建议整理为项目 / 专题
- 某类内容累积较多，建议统一处理

### 为什么 Review 必须放进 Phase 2

因为 PRD 和 MVP 文档已经写得很明确了：

Atlax 不是“自动归档工具”，而是“让知识开始替你工作”的系统。

如果 Demo 只做到输入、建议、归档，那它更像半个整理助手；

只有 Review 出来，用户才会第一次真正看到产品的 higher-level value。

### 退出标准 / 子阶段验收标准

- Weekly Review 页可正常打开
- 能看到本周统计数据
- 能看到至少一条健康信号或提醒卡片
- Review 内容与实际系统数据有对应关系，不是纯静态摆拍

---

## Phase 2.7｜Demo 演示路径封板与样例数据完善

### 阶段目标

把整个 Demo 变成可稳定演示、可讲故事、可让别人快速看懂的产品原型。

### 需要完成的内容

- 整理一套标准 Demo 演示路径
- 准备样例数据，避免空库演示
- 核心文案统一
- 核心页面空状态、无数据状态、异常提示补齐
- 对最容易翻车的流程进行手工回归

### 推荐演示路径

1. 用户输入一段想法 / 任务 / 会议记录
2. 内容进入 Inbox
3. 系统给出 type / tag / project 建议
4. 用户接受或微调后归档
5. 内容进入 Database View
6. 在 Project / Task / Entry 视图中重新找到内容
7. 打开 Weekly Review
8. 看到本周新增、整理率、项目动态和提醒
9. 完成一次“系统接住内容并推动下一步”的演示

### 退出标准 / 子阶段验收标准

- Demo 路径可从头到尾跑通
- 不依赖开发者口头解释也能大致看懂
- 空状态、无数据状态不会让演示中断
- 演示数据足够体现产品价值，而不是只有 1 条测试数据

## 四、Phase 2 总验收标准

Phase 2 只有在以下条件同时满足时，才能视为阶段完成。

### 1. 功能闭环验收

必须跑通：

- Capture
- Inbox Suggestion
- Tag 辅助整理
- Archive 入库
- Database View
- Weekly Review

### 2. 产品表达验收

必须做到：

- 界面已经像一个生产力工具，而不是 demo 表单
- 用户进入系统后能理解主界面结构
- 关键操作不需要频繁跨页面跳转
- 至少具备可演示的统一视觉和工作流

### 3. 数据闭环验收

必须保证：

- 输入内容不丢失
- 建议结果能进入结构化存储
- tag / type / project / task 的最小关系成立
- Review 页面数据来源于真实数据，而不是纯 mock

### 4. Demo 可演示验收

必须具备：

- 一条完整演示路径
- 一组高质量样例数据
- 页面空状态和错误提示不至于让演示崩掉
- 非开发者也能大致理解产品要解决什么问题

## 五、Phase 2 暂时砍掉 / 不在本阶段强行实现的能力

这些能力不是不要，而是**Phase 2 先记录、先预留入口、暂不强行做完**。

### 1. LLM 驱动整理

暂不接入。

原因：当前先验证规则引擎 + tag 双轨机制是否足够支撑主线体验。

后续可在体验成熟后作为增强层接入。

### 2. 自动 Links / 自动知识图谱

暂不在 Demo 强实现。

原因：展示价值高，但当前主线还没有足够稳。

后续可基于 tag / project / keyword similarity 逐步增加建议能力。

### 3. 树形知识结构的完整实现

暂不在 Phase 2 做完整知识树。

但必须在数据模型上保留扩展方向：

- 用户为 root
- tag 为一层组织节点
- entry / doc / idea / task 向 tag 与 project 聚合
- 后续再逐步扩展“树 + 网络混合”的知识组织方式

### 4. 导入功能

暂不做正式导入能力。

但必须记录为后续重点能力，并预留入口。

后续导入逻辑可基于：

- 用户已有 tag
- 系统建议 tag
- 结构化整理规则
    
    将外部内容纳入内部组织树。
    

### 5. 高级搜索

先不做复杂搜索系统。

Phase 2 以基础筛选和基础定位为主，后续 Phase 4 再判断是否前置。

### 6. 重型编辑器 / 高级 Markdown 编辑

先不做。

当前只要能完成记录、归档、查看和最小编辑即可。

## 六、Phase 2 结束后的自然衔接

如果 Phase 2 成功，后续阶段应自然转入：

### Phase 3｜质量收敛

重点解决：

- 架构收敛
- 测试与回归
- 状态一致性
- 页面稳定性
- 性能与维护成本

### Phase 4｜体验打磨与扩展能力

重点考虑：

- 界面细节优化
- tag 体验强化
- links 建议
- 树形知识组织增强
- 导入能力设计
- 搜索与复用能力增强
- 更强的 Review 与 Insight 表达
