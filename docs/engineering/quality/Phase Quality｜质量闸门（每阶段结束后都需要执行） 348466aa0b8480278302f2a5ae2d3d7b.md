## 阶段定位

本阶段不是一个独立的产品功能阶段，而是每轮功能开发完成后都必须执行的一次固定质量闸门。

它的目标不是新增产品能力，而是确保当前阶段产出的代码、目录结构、性能表现、工程规范与仓库卫生达到“可继续迭代”的标准。

Phase Quality 必须穿插在每轮开发之间执行：

- Phase 2 完成后，进入一次 Phase Quality
- Phase 3 完成后，再进入一次 Phase Quality
- 后续每轮新功能阶段结束后，都必须执行一次 Phase Quality

如果某一轮未通过 Phase Quality，则该轮开发不能视为真正完成，也不应继续进入下一阶段主开发。

---

## 阶段目标

Phase Quality 的目标是围绕以下原则，对当前代码进行系统性收敛：

1. 保证仓库干净清爽，不保留冗余、重复、废弃、无意义的代码与文件
2. 保证项目结构清晰、职责分明，便于后续维护与扩展
3. 保证每轮开发结束后都进行性能检查，避免明显低效实现累积成长期维护成本
4. 严格遵守 Clean Code 原则，以清晰命名、单一职责、低耦合、可读性优先为核心标准
5. 建立最小但真实可执行的质量基线，确保项目不是“能跑一次”，而是“能稳定继续开发”

---

## 阶段拆分

### 3.1 仓库卫生检查

### 子阶段目标

清理本轮开发过程中遗留的冗余、无用、重复、临时、污染性内容，确保仓库始终保持可交付、可阅读、可维护状态。

### 检查范围

- 冗余文件
- 重复文件
- 临时调试文件
- 构建产物
- 缓存目录
- 本地系统垃圾文件
- 无意义占位文件
- 已废弃但未移除的旧代码
- 与当前阶段无关的残留提交内容

### 子阶段验收标准

- 仓库内不存在明显无用、重复、废弃但未清理的文件
- `.gitignore` 能有效排除构建产物、缓存、系统垃圾
- 当前分支改动边界清晰，不混入大量无关脏文件
- 没有“暂时先留着以后再删”的长期垃圾代码
- 目录与文档状态基本一致，不出现“文档一个世界、代码一个世界”的情况

---

### 3.2 项目结构收敛检查

### 子阶段目标

确保项目目录结构清晰、职责明确，不同工程内容按照责任边界合理归位，并在结构优化后保持项目可正常运行。

### 检查范围

- 页面层代码
- 领域逻辑代码
- 持久化适配代码
- 静态资源与素材文件
- 前后端目录边界
- 未来扩展入口目录
- 文档与工程结构映射关系

### 子阶段验收标准

- 页面展示逻辑、领域逻辑、持久化逻辑不混放
- 前端代码与未来后端/扩展入口边界清晰
- 静态素材文件独立存放，不散落在业务目录中
- 目录命名能体现真实职责，不依赖个人记忆理解结构
- 结构调整后项目可正常运行，无新增功能性 bug
- 当前结构便于后续继续维护，而不是“暂时能跑但越来越乱”

---

### 3.3 性能与实现效率检查

### 子阶段目标

在当前单用户、本地优先、Web-first MVP 的前提下，对当前实现进行最小但必要的性能检查，排除明显低效路径，降低后续维护与扩展成本。

### 检查范围

- 重复渲染
- 重复查询
- render 中重复 filter / find / map
- 全量刷新替代局部更新
- 不必要的状态更新
- 明显低效的数据处理逻辑
- 容易放大为未来瓶颈的实现方式

### 子阶段验收标准

- 当前主闭环不存在明显高频低效代码路径
- 关键交互不依赖无脑全量刷新维持正确性
- 重复计算逻辑已收敛到 selector、工具函数或明确处理层
- 当前实现不会明显阻碍后续接入缓存、高性能数据库或更高性能执行环境
- 性能优化以清理低质量实现为目标，不引入过度设计与提前抽象

---

### 3.4 Clean Code 与代码规范检查

### 子阶段目标

确保当前代码符合统一、严格、可执行的 Clean Code 标准，防止随着代码量增长快速积累脏代码和维护成本。

### 检查范围

- 命名清晰度
- 函数职责单一性
- 模块边界清晰度
- 组件复杂度
- 重复代码
- 隐式逻辑
- any、非空断言、硬编码的使用情况
- 错误处理与副作用位置
- 可读性与可解释性

### 子阶段验收标准

- 命名能够准确表达真实职责，不使用模糊缩写或临时命名
- 函数、组件、模块职责单一，不出现明显“上帝函数 / 上帝组件”
- 重复逻辑得到收敛，不允许复制粘贴式扩展
- 避免 any 扩散，避免滥用非空断言，避免隐式副作用
- 代码优先追求可读、可维护、可追踪，而不是炫技抽象
- 当前阶段新增代码与重构代码符合统一规范，不出现“每轮一个写法”的情况

---

### 3.5 质量基线与回归验证检查

### 子阶段目标

为每轮开发建立统一的最小质量门，确保代码在进入下一阶段前已经通过基础校验与回归验证。

### 检查范围

- lint
- typecheck
- test
- build
- 主闭环手工验证
- 本轮新增能力验证
- 异常路径与状态流验证

### 子阶段验收标准

- 每轮开发结束后必须通过 lint / typecheck / test / build
- 主闭环可被手工稳定回归验证
- 本轮新增状态变化具备明确验证路径
- 未通过质量门的代码不得进入下一阶段
- 本轮必须输出一份简短质检结论，说明：
    - 已修复问题
    - 延后处理问题
    - 当前可接受技术债
    - 是否允许进入下一阶段

---

## 阶段总验收标准

每一次 Phase Quality 执行完成后，必须同时满足以下条件：

1. 仓库保持干净，没有明显冗余、重复、废弃、污染性文件
2. 项目结构清晰，职责边界比进入本轮前更明确
3. 当前实现不存在明显低效、重复、难维护的高风险路径
4. 当前代码符合统一的 Clean Code 标准，命名、职责、边界、可读性可接受
5. lint / typecheck / test / build 全部通过
6. 主闭环与本轮新增能力已完成回归验证
7. 已形成一份本轮质量结论，明确当前代码是否允许进入下一阶段

---

## 执行规则

### 规则 1

Phase Quality 不可跳过。任何功能阶段完成后，都必须执行一次 Phase Quality。

### 规则 2

Phase Quality 不以新增功能数量作为产出，而以“当前成果是否值得保留并继续迭代”作为产出。

### 规则 3

若未通过 Phase Quality，则该轮功能开发不得标记完成。

### 规则 4

Phase Quality 必须留下可追溯的质检记录，至少包含：

- 仓库卫生结论
- 结构收敛结论
- 性能检查结论
- Clean Code 检查结论
- 回归验证结论
- 是否准许进入下一阶段

---

## 本阶段关注重点

Phase Quality 关注的是：

- 干净
- 清晰
- 稳定
- 可维护
- 可继续开发

而不是：

- 新增多少页面
- 再堆多少功能
- 提前做复杂架构
- 为未来十倍规模做过早设计

---

## 一句话定义本阶段成果

**Phase Quality 的完成标志，不是“代码还能跑”，而是“这批代码值得继续留在项目里”。**
